# Error Handling

No application is immune to errors.

Database connections may fail, external APIs can become unavailable, users may submit invalid input, and unexpected exceptions can occur at runtime.

A robust error handling system ensures that these failures are:

- Detected quickly
- Reported consistently
- Logged appropriately
- Presented safely to users
- Managed without crashing the application

Bejibun provides a centralized error handling system that allows applications to respond gracefully when something goes wrong.

---

# What Is Error Handling?

Error handling is the process of detecting, processing, and responding to failures that occur during application execution.

Without proper error handling:

```text
Application Error
       │
       ▼
Application Crash
```

With proper error handling:

```text
Application Error
       │
       ▼
Exception Handler
       │
       ▼
Log Error
       │
       ▼
Generate Response
       │
       ▼
Client
```

The goal is not to eliminate errors entirely, but to manage them predictably.

---

# The Error Lifecycle

When an exception occurs, Bejibun follows a structured process.

```text
Exception Thrown
        │
        ▼
Exception Captured
        │
        ▼
Exception Handler
        │
        ▼
Logging
        │
        ▼
Response Generation
        │
        ▼
Client
```

This centralized workflow keeps error handling consistent across the application.

# Types of Errors

Applications typically encounter several categories of errors.

---

## Validation Errors

Occur when incoming data fails validation rules.

Example:

```json
{
  "email": "invalid-email"
}
```

Response:

```json
{
  "errors": [
    {
      "field": "email",
      "message": "Invalid email address"
    }
  ]
}
```

Validation errors are expected and should not be treated as application failures.

---

## Authentication Errors

Occur when a user is not authenticated.

Example:

```http
401 Unauthorized
```

Response:

```json
{
  "message": "Unauthorized"
}
```

---

## Authorization Errors

Occur when a user lacks permission to perform an action.

Example:

```http
403 Forbidden
```

Response:

```json
{
  "message": "Forbidden"
}
```

---

## Resource Not Found Errors

Occur when a requested resource does not exist.

Example:

```http
GET /users/999
```

Response:

```http
404 Not Found
```

---

## Database Errors

Occur during database operations.

Examples:

```text
Connection Failure
Query Error
Constraint Violation
Transaction Failure
```

These errors should generally be logged and handled centrally.

---

## External Service Errors

Occur when communicating with external systems.

Examples:

```text
Payment Provider
Email Service
Cloud Storage
Third-Party API
```

Applications should expect these services to fail occasionally.

---

## Application Exceptions

Occur when unexpected conditions arise within application logic.

Example:

```ts
throw new Error(
  "Unexpected state"
);
```

These typically indicate bugs or unforeseen edge cases.

---

# Throwing Exceptions

Exceptions can be raised when an operation cannot continue safely.

Example:

```ts
throw new Error(
  "User not found"
);
```

Execution stops immediately.

```text
Code
 │
 ▼
Exception
 │
 ▼
Handler
```

This prevents invalid application states.

---

# Built-In Exceptions

Framework components may throw specialized exceptions.

Examples:

```text
ValidationException

AuthenticationException

AuthorizationException

NotFoundException
```

These exceptions allow Bejibun to generate meaningful responses automatically.

---

# Creating Custom Exceptions

Applications can define domain-specific exceptions.

Example:

```ts
export class PaymentFailedException
  extends Error {

  constructor() {
    super(
      "Payment failed"
    );
  }
}
```

Usage:

```ts
throw new PaymentFailedException();
```

Custom exceptions improve clarity and maintainability.

---

# Catching Exceptions

Sometimes exceptions should be handled locally.

Example:

```ts
try {
  await processPayment();
} catch (error) {
  Logger.error(
    error.message
  );
}
```

This allows recovery from expected failures.

---

# When to Catch Exceptions

Catch exceptions when:

- Recovery is possible
- Alternative behavior exists
- Additional context must be added

Example:

```ts
try {
  return await gateway.charge();
} catch {
  return fallbackGateway.charge();
}
```

---

# When Not to Catch Exceptions

Avoid catching exceptions simply to ignore them.

Bad:

```ts
try {
  await process();
} catch {}
```

Silent failures make debugging difficult.

If an exception is caught, it should usually be:

- Logged
- Transformed
- Handled appropriately

---

# Global Exception Handler

Bejibun provides a centralized exception handler.

Example structure:

```text
app/
└── Exceptions/
    └── Handler.ts
```

Responsibilities:

- Logging exceptions
- Formatting responses
- Transforming errors
- Reporting critical failures

Centralized handling keeps behavior consistent.

---

# Reporting Exceptions

Errors should be reported before responding to clients.

Example:

```ts
Logger.error(
  error.message
);
```

Or:

```ts
Logger.error(
  "Database failure",
  {
    error
  }
);
```

Logging provides operational visibility.

---

# Rendering Exceptions

The exception handler converts exceptions into HTTP responses.

Example:

```ts
render(error) {
  return response.internalServerError({
    message:
      "Something went wrong"
  });
}
```

Users receive a safe response while internal details remain protected.

---

# Environment-Aware Error Responses

Development and production environments often handle errors differently.

Development:

```json
{
  "message": "Database connection failed",
  "stack": "..."
}
```

Production:

```json
{
  "message": "Internal Server Error"
}
```

Configuration:

```env
APP_DEBUG=true
```

Development environments benefit from detailed diagnostics.

Production environments prioritize security.

---

# HTTP Exception Responses

Bejibun may automatically convert common exceptions into HTTP responses.

Example mapping:

| Exception | HTTP Status |
|------------|------------|
| ValidationException | 422 |
| AuthenticationException | 401 |
| AuthorizationException | 403 |
| NotFoundException | 404 |
| Generic Error | 500 |

This creates a consistent API experience.

---

# Validation Error Handling

Validation failures should return actionable feedback.

Example:

```json
{
  "errors": [
    {
      "field": "email",
      "message": "The email field is required."
    }
  ]
}
```

Applications should never treat validation failures as system errors.

---

# API Error Responses

A consistent response structure improves developer experience.

Example:

```json
{
  "message": "Resource not found"
}
```

Or:

```json
{
  "message": "Validation failed",
  "errors": []
}
```

Consistency makes APIs easier to consume.

---

# Error Logging

Every unexpected exception should be logged.

Example:

```ts
Logger.error(
  "Unhandled exception",
  {
    error
  }
);
```

Logs should include:

- Message
- Stack trace
- Context
- Request metadata

---

# Request Context

Errors become easier to diagnose when request information is available.

Example:

```ts
Logger.error(
  "Order creation failed",
  {
    userId,
    orderId,
    route
  }
);
```

Context dramatically reduces debugging time.

---

# Security Considerations

Error responses should never expose sensitive information.

Avoid:

```json
{
  "databasePassword": "...",
  "query": "...",
  "stack": "..."
}
```

in production environments.

Users should only receive information necessary to understand the failure.

---

# Handling External Service Failures

External systems should be treated as unreliable.

Example:

```ts
try {
  await mail.send();
} catch {
  Logger.warning(
    "Mail delivery failed"
  );
}
```

Applications should degrade gracefully whenever possible.

---

# Handling Database Failures

Example:

```ts
try {
  await User.create(data);
} catch (error) {
  Logger.error(
    "Database operation failed",
    { error }
  );

  throw error;
}
```

Database failures are often critical and should be logged consistently.

```text
Application Error
        │
        ▼

        ▼

        │
```

# Best Practices

### Fail Fast

Detect problems as early as possible.

---

### Use Specific Exceptions

Good:

```ts
throw new UserNotFoundException();
```

Avoid:

```ts
throw new Error();
```

when a more descriptive exception exists.

---

### Log Unexpected Failures

Unhandled exceptions should always be logged.

---

### Protect Sensitive Information

Never expose internal implementation details to users.

---

### Keep Responses Consistent

Error responses should follow a predictable structure.

---

# Common Mistakes

### Swallowing Exceptions

Avoid:

```ts
catch {}
```

This hides failures and complicates debugging.

---

### Exposing Stack Traces

Avoid returning stack traces in production.

---

### Logging Sensitive Data

Never log:

```text
Passwords
API Keys
Access Tokens
Payment Information
```

---

### Using Exceptions for Normal Control Flow

Exceptions should represent exceptional situations, not regular application behavior.
