# Dependency Injection

Dependency Injection (DI) is one of the most important architectural concepts in modern application development.

It allows classes to receive their dependencies from the framework rather than creating them manually.

Instead of a class being responsible for constructing everything it needs, the framework provides those dependencies automatically through the Service Container.

This results in applications that are:

- Easier to maintain
- Easier to test
- More flexible
- Less tightly coupled
- More scalable

---

# What Is a Dependency?

A dependency is any object that another object requires to perform its work.

Example:

```ts
export default class UserService {
  constructor(
    protected logger: Logger
  ) {}
}
```

In this example:

```text
UserService
     │
     ▼
Logger
```

`Logger` is a dependency of `UserService`.

Without a logger, the service may not function correctly.

---

# The Problem Without Dependency Injection

Consider the following example:

```ts
export default class UserService {
  protected logger =
    new Logger();
}
```

At first glance, this seems simple.

However, several problems quickly emerge:

### Tight Coupling

`UserService` is permanently tied to `Logger`.

Replacing the logger requires modifying the service.

---

### Difficult Testing

Testing becomes harder because dependencies are created internally.

Example:

```ts
const service =
  new UserService();
```

You cannot easily replace the logger with a mock implementation.

---

### Reduced Flexibility

Switching implementations becomes cumbersome.

Example:

```text
Logger
   ▼
FileLogger
```

Every class using the logger may require changes.

---

# The Dependency Injection Solution

Instead of creating dependencies manually:

```ts
export default class UserService {
  protected logger =
    new Logger();
}
```

Inject them:

```ts
export default class UserService {
  constructor(
    protected logger: Logger
  ) {}
}
```

Now the service no longer cares where the logger comes from.

The framework provides it automatically.

---

# How Dependency Injection Works

The process is straightforward.

```text
Application
      │
      ▼
Service Container
      │
      ▼
Resolve Dependencies
      │
      ▼
Instantiate Class
```

Example:

```ts
Container.make(
  UserService
);
```

The container:

1. Inspects the constructor.
2. Finds required dependencies.
3. Resolves them.
4. Creates the object.
5. Returns the fully configured instance.

---

# Constructor Injection

Constructor Injection is the preferred form of dependency injection in Bejibun.

Example:

```ts
export default class UserService {
  constructor(
    protected logger: Logger,
    protected cache: Cache
  ) {}
}
```

Dependency graph:

```text
UserService
 ├── Logger
 └── Cache
```

The framework automatically resolves both dependencies.

---

# Automatic Resolution

Suppose you have:

```ts
class Logger {}

class UserService {
  constructor(
    private logger: Logger
  ) {}
}
```

Resolving:

```ts
Container.make(
  UserService
);
```

Internally becomes:

```ts
new UserService(
  new Logger()
);
```

The difference is that the framework performs the work for you.

---

# Nested Dependency Resolution

Dependencies often depend on other dependencies.

Example:

```ts
class Logger {}

class UserRepository {
  constructor(
    private logger: Logger
  ) {}
}

class UserService {
  constructor(
    private repository:
      UserRepository
  ) {}
}
```

Resolution tree:

```text
UserService
      │
      ▼
UserRepository
      │
      ▼
Logger
```

The container automatically resolves the entire chain.

---

# Controller Injection

Dependency injection is commonly used within controllers.

Example:

```ts
export default class UserController {
  constructor(
    protected users:
      UserService
  ) {}
}
```

Route:

```ts
Router.get(
  "/users",
  UserController
);
```

When the route is executed, the framework creates the controller and injects the required service.

---

# Injecting Multiple Services

Controllers may receive several dependencies.

Example:

```ts
export default class UserController {
  constructor(
    protected users:
      UserService,

    protected mail:
      MailService,

    protected cache:
      CacheService
  ) {}
}
```

The framework resolves all dependencies automatically.

---

# Service Injection

Services can also receive dependencies.

Example:

```ts
export default class UserService {
  constructor(
    protected repository:
      UserRepository,

    protected logger:
      Logger
  ) {}
}
```

This creates a clean separation between layers.

---

# Repository Injection

For larger applications:

```text
Controller
      │
      ▼
Service
      │
      ▼
Repository
      │
      ▼
Database
```

Example:

```ts
class UserRepository {}

class UserService {
  constructor(
    private repository:
      UserRepository
  ) {}
}
```

This architecture promotes maintainability and scalability.

---

# Interface-Based Injection

Applications should often depend on contracts rather than implementations.

Example contract:

```ts
export interface Mailer {
  send(): Promise<void>;
}
```

Implementation:

```ts
export class SmtpMailer
  implements Mailer {}
```

Container binding:

```ts
Container.bind(
  "mailer",
  () => new SmtpMailer()
);
```

Usage:

```ts
constructor(
  protected mailer: Mailer
) {}
```

Benefits:

- Swappable implementations
- Easier testing
- Better architecture

---

# Request Injection

Framework-provided objects may also be injected.

Example:

```ts
export default class UserController {
  async store(
    request: Request
  ) {
    //
  }
}
```

Possible framework injections include:

- Request
- Response
- Auth
- Session
- Context

This eliminates the need for global state.

---

# Method Injection

Dependencies may also be injected directly into methods.

Example:

```ts
export default class UserController {
  async index(
    service: UserService
  ) {
    return service.all();
  }
}
```

The framework resolves the dependency before executing the method.

---

# Dependency Injection vs Service Locator

Both patterns use the Service Container, but they are not the same.

---

## Dependency Injection

Good:

```ts
constructor(
  private logger: Logger
) {}
```

Dependencies are explicit.

You can immediately see what the class requires.

---

## Service Locator

Less ideal:

```ts
const logger =
  Container.make(
    Logger
  );
```

Dependencies are hidden inside the implementation.

This can make testing and maintenance more difficult.

---

# Testing with Dependency Injection

Dependency injection greatly simplifies testing.

Example:

```ts
class MockLogger {
  info() {}
}
```

Inject:

```ts
const service =
  new UserService(
    new MockLogger()
  );
```

Benefits:

- No container required
- No database required
- No external services required
- Faster tests

Testing becomes significantly easier.

# Best Practices

### Prefer Constructor Injection

Good:

```ts
constructor(
  private logger: Logger
) {}
```

Avoid:

```ts
this.logger =
  Container.make(
    Logger
  );
```

inside classes.

---

### Keep Dependencies Explicit

A constructor should clearly communicate what the class requires.

Good:

```ts
constructor(
  logger: Logger
) {}
```

Avoid hidden dependencies.

---

### Depend on Contracts

Prefer:

```ts
Mailer
```

instead of:

```ts
SmtpMailer
```

whenever practical.

---

### Keep Constructors Reasonable

Avoid:

```ts
constructor(
  a,
  b,
  c,
  d,
  e,
  f,
  g
) {}
```

Too many dependencies may indicate that a class has too many responsibilities.

---

### Favor Composition

Build applications from smaller services rather than large monolithic classes.

---

# Common Mistakes

### Creating Dependencies Manually

Avoid:

```ts
new Logger();
```

inside application services.

---

### Overusing the Container

Avoid:

```ts
Container.make(...)
```

everywhere.

Inject dependencies instead.

---

### Circular Dependencies

Avoid:

```text
UserService
     │
     ▼
MailService
     │
     ▼
UserService
```

Circular dependencies complicate resolution and maintenance.

---

### Treating DI as Magic

Dependency injection is simply object construction delegated to the framework.

Understanding this principle makes debugging much easier.
