# Service Container

The Service Container is the heart of dependency management in Bejibun.

It is responsible for creating, storing, and resolving application services whenever they are needed.

Instead of manually constructing dependencies throughout your application, the container handles object creation automatically, allowing your code to remain clean, testable, and maintainable.

---

# What Is a Service Container?

A Service Container is an object registry responsible for managing application dependencies.

Without a container:

```ts
const logger = new Logger();

const userService =
  new UserService(logger);

const controller =
  new UserController(
    userService
  );
```

As applications grow, manually creating dependencies becomes difficult to manage.

With a container:

```ts
const controller =
  Container.make(
    UserController
  );
```

The framework automatically resolves all required dependencies.

---

# Why Use a Service Container?

The Service Container provides several benefits:

- Dependency Injection
- Automatic Resolution
- Loose Coupling
- Improved Testing
- Centralized Service Registration
- Better Maintainability

Instead of depending on implementations directly, classes depend on abstractions.

---

# How the Container Works

At a high level, the container performs three tasks:

```text
Register Service
       │
       ▼
Store Definition
       │
       ▼
Resolve Service
```

Example:

```text
Container
 ├── Logger
 ├── Cache
 ├── Mail
 └── UserService
```

When a dependency is requested, the container returns the appropriate instance.

---

# Service Registration

Before a service can be resolved, it must be registered.

Example:

```ts
Container.bind(
  "logger",
  () => new Logger()
);
```

Now the service can be resolved anywhere in the application.

```ts
const logger =
  Container.make("logger");
```

---

# Binding Classes

You may also bind classes directly.

Example:

```ts
Container.bind(
  UserService,
  () => new UserService()
);
```

Resolve:

```ts
const service =
  Container.make(
    UserService
  );
```

The container returns an instance of the registered class.

---

# Singleton Bindings

Some services should only exist once during the application's lifecycle.

Examples:

- Database Connections
- Cache Managers
- Event Dispatchers
- Loggers

For these cases, use a singleton.

```ts
Container.singleton(
  Logger,
  () => new Logger()
);
```

Now every resolution returns the same instance.

```ts
const a =
  Container.make(Logger);

const b =
  Container.make(Logger);

a === b; // true
```

---

# Transient Bindings

By default, services may be created each time they are requested.

Example:

```ts
Container.bind(
  UserService,
  () => new UserService()
);
```

Each resolution creates a new instance.

```ts
const a =
  Container.make(
    UserService
  );

const b =
  Container.make(
    UserService
  );

a === b; // false
```

Use transient bindings when services should remain isolated.

---

# Dependency Injection

The most common use of the Service Container is dependency injection.

Instead of creating dependencies manually:

```ts
export default class UserController {
  constructor() {
    this.service =
      new UserService();
  }
}
```

Inject them:

```ts
export default class UserController {
  constructor(
    protected service:
      UserService
  ) {}
}
```

The container automatically resolves the dependency.

---

# Constructor Injection

Constructor injection is the preferred approach.

Example:

```ts
export default class UserService {
  constructor(
    protected logger:
      Logger
  ) {}
}
```

Lifecycle:

```text
UserService
     │
     ▼
Logger Dependency
     │
     ▼
Container Resolution
```

The framework automatically provides the required dependencies.

---

# Nested Dependencies

The container can resolve dependency chains.

Example:

```ts
UserController
     │
     ▼
UserService
     │
     ▼
Logger
```

Resolution:

```text
Container
     │
     ▼
Logger
     │
     ▼
UserService
     │
     ▼
UserController
```

You only request the controller—the framework resolves everything else.

---

# Resolving Services Manually

Sometimes a service must be resolved directly.

Example:

```ts
const cache =
  Container.make(
    CacheService
  );
```

This is useful for:

- CLI commands
- Bootstrap code
- Framework internals

However, dependency injection should generally be preferred.

---

# Service Providers

Service registration is commonly performed within service providers.

Example:

```text
app/
└── Providers/
    └── AppProvider.ts
```

Example:

```ts
export default class AppProvider {
  register() {
    Container.singleton(
      Logger,
      () => new Logger()
    );
  }
}
```

Service providers centralize application bootstrapping.

---

# Register vs Boot

Service providers often contain two lifecycle methods.

Example:

```ts
export default class AppProvider {
  register() {}

  boot() {}
}
```

### Register

Used for:

- Binding services
- Registering dependencies
- Container setup

Example:

```ts
register() {
  Container.bind(
    UserService,
    () => new UserService()
  );
}
```

---

### Boot

Used after all services have been registered.

Example:

```ts
boot() {
  //
}
```

Typical responsibilities:

- Event listeners
- Route registration
- Framework initialization

---

# Binding Interfaces

Applications should often depend on contracts rather than concrete implementations.

Example contract:

```ts
export interface Mailer {
  send(): Promise<void>;
}
```

Implementation:

```ts
export class SmtpMailer
  implements Mailer {}
```

Binding:

```ts
Container.bind(
  "mailer",
  () => new SmtpMailer()
);
```

Resolution:

```ts
const mailer =
  Container.make("mailer");
```

This allows implementations to be swapped without changing application code.

---

# Practical Example

Suppose you have:

```ts
class Logger {}

class UserService {
  constructor(
    private logger: Logger
  ) {}
}

class UserController {
  constructor(
    private users:
      UserService
  ) {}
}
```

Request:

```ts
Container.make(
  UserController
);
```

Resolution process:

```text
UserController
       │
       ▼
UserService
       │
       ▼
Logger
```

The container automatically builds the dependency graph.

# Testing Benefits

Dependency injection makes testing significantly easier.

Without a container:

```ts
new UserService(
  new Logger()
);
```

With dependency injection:

```ts
new UserService(
  mockLogger
);
```

Example:

```ts
const mockLogger = {
  info() {}
};
```

Benefits:

- Faster tests
- Easier mocking
- Better isolation
- More predictable behavior

---

# Container Events

Advanced applications may need to react when services are resolved.

Example:

```ts
Container.resolving(
  Logger,
  logger => {
    //
  }
);
```

Possible use cases:

- Instrumentation
- Monitoring
- Framework extensions

# Service Lifetimes

Services generally fall into three categories.

| Lifetime | Description |
|-----------|-------------|
| Singleton | One instance for the application |
| Scoped | One instance per request |
| Transient | New instance every resolution |

Example:

```text
Singleton
 └── Database

Scoped
 └── Request Context

Transient
 └── UserService
```

Choosing the correct lifetime improves performance and predictability.

# Best Practices

### Prefer Constructor Injection

Good:

```ts
constructor(
  private logger: Logger
) {}
```

Avoid:

```ts
const logger =
  Container.make(
    Logger
  );
```

inside application code.

---

### Register Services Centrally

Use service providers instead of scattered registrations.

---

### Depend on Contracts

Prefer abstractions over concrete implementations whenever possible.

---

### Use Singletons Carefully

Singletons are ideal for shared infrastructure services but should not hold request-specific state.

---

### Keep Services Focused

Each service should have a single responsibility.

---

# Common Mistakes

### Manually Creating Dependencies

Avoid:

```ts
new UserService(
  new Logger()
);
```

throughout the application.

---

### Service Locator Abuse

Avoid excessive use of:

```ts
Container.make(...)
```

inside business logic.

Dependency injection is usually cleaner.

---

### Registering Too Much in Controllers

Controllers should consume services, not configure them.

---

### Circular Dependencies

Avoid:

```text
UserService
     │
     ▼
MailService
     │
     ▼
UserService
```

Circular dependency chains make applications difficult to maintain.
