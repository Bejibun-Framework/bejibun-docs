# Configuration System

Every application requires configuration.

Database connections, cache drivers, authentication providers, storage disks, third-party integrations, and application settings all rely on configuration values.

The Bejibun Configuration System provides a centralized way to define, organize, and access configuration throughout your application.

Instead of scattering configuration across your codebase, Bejibun keeps it structured, predictable, and environment-aware.

---

# What Is the Configuration System?

The Configuration System is responsible for:

- Loading configuration files
- Reading environment variables
- Providing configuration values
- Managing application settings
- Supporting multiple environments
- Enabling future configuration caching

At a high level:

```text
Environment Variables
         │
         ▼
Configuration Files
         │
         ▼
Configuration Service
         │
         ▼
Application Code
```

This separation allows application logic to remain independent from deployment-specific configuration.

---

# Configuration Philosophy

Configuration should be:

### Centralized

All application settings should live in dedicated configuration files.

---

### Environment-Aware

Development, testing, staging, and production environments often require different values.

---

### Type-Safe

Configuration should provide predictable and validated values.

---

### Easy to Maintain

Changes should be made in one place rather than throughout the application.

---

# Configuration Directory

All configuration files are stored inside:

```text
config/
```

Example:

```text
config/
├── app.ts
├── auth.ts
├── cache.ts
├── database.ts
├── mail.ts
├── queue.ts
├── services.ts
└── storage.ts
```

Each file is responsible for a single area of configuration.

---

# Configuration Files

A configuration file exports a plain object.

Example:

```ts
export default {
  name: "Bejibun",

  timezone: "UTC",

  debug: true
};
```

File:

```text
config/app.ts
```

The framework automatically loads configuration files during application startup.

---

# Configuration Namespaces

Each configuration file creates its own namespace.

Example:

```ts
// config/app.ts

export default {
  name: "Bejibun"
};
```

Access:

```ts
Config.get("app.name");
```

Result:

```text
Bejibun
```

The filename becomes the root namespace.

---

# Accessing Configuration

Configuration values can be retrieved through the Config service.

Example:

```ts
Config.get("app.name");
```

Result:

```text
Bejibun
```

Nested values are also supported.

Example:

```ts
Config.get(
  "database.connections.mysql.host"
);
```

---

# Default Values

You may provide a fallback value.

Example:

```ts
Config.get(
  "mail.default",
  "smtp"
);
```

If the key does not exist:

```text
smtp
```

is returned.

This helps prevent runtime failures caused by missing configuration.

---

# Nested Configuration

Configuration objects can be deeply nested.

Example:

```ts
export default {
  connections: {
    mysql: {
      host: "localhost",

      port: 3306
    }
  }
};
```

Access:

```ts
Config.get(
  "database.connections.mysql.host"
);
```

Result:

```text
localhost
```

Dot notation makes deeply nested configuration easy to access.

---

# Environment Variable Integration

Configuration files often rely on environment variables.

Example:

```ts
export default {
  host: Env.get(
    "DB_HOST",
    "127.0.0.1"
  ),

  port: Env.get(
    "DB_PORT",
    3306
  )
};
```

This allows configuration to change without modifying source code.

---

# Application Configuration

Example:

```ts
// config/app.ts

export default {
  name: Env.get(
    "APP_NAME",
    "Bejibun"
  ),

  env: Env.get(
    "APP_ENV",
    "development"
  ),

  debug: Env.get(
    "APP_DEBUG",
    true
  )
};
```

Usage:

```ts
Config.get("app.name");
```

---

# Database Configuration

Example:

```ts
// config/database.ts

export default {
  default: "mysql",

  connections: {
    mysql: {
      host: Env.get("DB_HOST"),

      database:
        Env.get("DB_DATABASE")
    }
  }
};
```

Usage:

```ts
Config.get(
  "database.default"
);
```

---

# Service Configuration

External services should have dedicated configuration files.

Example:

```ts
// config/services.ts

export default {
  stripe: {
    secret:
      Env.get(
        "STRIPE_SECRET"
      )
  },

  mailgun: {
    apiKey:
      Env.get(
        "MAILGUN_API_KEY"
      )
  }
};
```

Usage:

```ts
Config.get(
  "services.stripe.secret"
);
```

# Runtime Configuration Access

Configuration can be accessed from:

- Controllers
- Services
- Middleware
- Jobs
- Event Listeners
- Console Commands

Example:

```ts
export default class MailService {
  send() {
    const host =
      Config.get(
        "mail.host"
      );
  }
}
```

Configuration remains available throughout the application lifecycle.

# Type-Safe Configuration

One advantage of TypeScript is stronger configuration validation.

Example:

```ts
export interface AppConfig {
  name: string;

  env: string;

  debug: boolean;
}
```

Benefits include:

- Autocomplete
- Compile-time validation
- Better IDE support
- Safer refactoring

Large applications should leverage TypeScript whenever possible.

# Configuration Loading Process

During application startup:

```text
Application Starts
         │
         ▼
Load Environment Variables
         │
         ▼
Load Configuration Files
         │
         ▼
Register Config Service
         │
         ▼
Application Ready
```

Configuration is loaded before routes, controllers, and services are executed.

---

# Configuration Immutability

Configuration should generally be treated as read-only.

Good:

```ts
Config.get("app.name");
```

Avoid:

```ts
Config.set(
  "app.name",
  "Something Else"
);
```

during runtime.

Configuration changes should typically occur through configuration files or environment variables.

---

# Organizing Large Applications

As applications grow, configuration should remain organized by domain.

Good:

```text
config/
├── app.ts
├── auth.ts
├── cache.ts
├── database.ts
├── mail.ts
├── queue.ts
└── storage.ts
```

Avoid:

```text
config/
└── settings.ts
```

containing hundreds of unrelated settings.

Smaller focused files are easier to maintain.

---

# Configuration Caching

Loading configuration files on every startup can become expensive in large applications.

Future versions of Bejibun may support configuration caching.

Example:

```bash
bun ace config:cache
```

Potential workflow:

```text
Config Files
      │
      ▼
Compiled Cache
      │
      ▼
Fast Startup
```

Benefits include:

- Faster boot times
- Reduced file system access
- Improved production performance

# Common Mistakes

### Accessing Environment Variables Everywhere

Avoid:

```ts
Env.get("DB_HOST");
```

throughout your application.

Prefer:

```ts
Config.get(
  "database.host"
);
```

This keeps environment concerns isolated.

---

### Hardcoding Configuration

Avoid:

```ts
const host =
  "localhost";
```

Use configuration files instead.

---

### Storing Secrets in Code

Avoid:

```ts
secret:
  "my-secret-key";
```

Use:

```ts
Env.get(
  "API_SECRET"
);
```

---

### Mixing Business Logic and Configuration

Configuration files should contain settings, not application logic.

---

# Best Practices

### Use Dedicated Configuration Files

Keep related settings together.

---

### Read Environment Variables Once

Load them inside configuration files.

---

### Prefer Config Over Env

Application code should usually access:

```ts
Config.get(...)
```

rather than:

```ts
Env.get(...)
```

---

### Provide Defaults

Reasonable defaults improve developer experience.

---

### Keep Configuration Predictable

Developers should always know where a setting belongs.
