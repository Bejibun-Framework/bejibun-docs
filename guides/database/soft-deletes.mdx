Bejibun models support **soft deletes** out of the box. Instead of removing a row, `delete()` stamps the `deleted_at` column, and soft-deleted rows are automatically excluded from normal queries.

## Requirements

Your table needs a nullable `deleted_at` column and the model needs the `deleted_at` field:

```ts
// migration
table.timestamp("deleted_at");
```

```ts
// model
declare deleted_at: NullableTimestamp;
```

(The default `make:model` and `make:migration` stubs include these.)

## Soft deleting

```ts
await TestModel.find(id).delete();   // sets deleted_at = now()
```

The row stays in the database but disappears from default queries like `TestModel.all()`.

## Restoring

Clear `deleted_at` to bring a record back:

```ts
await TestModel.find(id).restore();
```

## Permanent deletion

To remove a row for good, bypass soft deletes with `forceDelete()`:

```ts
await TestModel.find(id).forceDelete();
```

## Querying trashed records

By default queries hide soft-deleted rows. Include or isolate them with:

<CodeGroup>
```ts With trashed
const all = await TestModel.withTrashed();    // active + trashed
```

```ts Only trashed
const trashed = await TestModel.onlyTrashed(); // trashed only
```
</CodeGroup>

## Method reference

| Method | Effect |
| --- | --- |
| `.delete()` / `.del()` | Soft delete (set `deleted_at`) |
| `.forceDelete()` | Permanent delete |
| `.restore()` | Clear `deleted_at` on trashed rows |
| `.withTrashed()` | Include trashed rows |
| `.onlyTrashed()` | Only trashed rows |

## Validating against trashed rows

The `exists` validation rule can opt into trashed rows — useful for a restore endpoint that must accept an already-deleted ID:

```ts
id: super.validator.number().min(1).exists(TestModel, "id", true)
```

```ts app/controllers/TestController.ts
public async restore(request: Bun.BunRequest): Promise<Response> {
    const body = await super.parse(request);
    await super.validate(TestValidator.restore, body);

    const test = await TestModel.find(body.id as number | string).restore();

    return super.response.setData(test).send();
}
```

# Soft Deletes

Deleting data is often more complicated than it appears.

Consider a user account:

```text
Delete User
```

Should the application permanently remove:

- Posts
- Orders
- Audit Logs
- Settings
- Activity History

In many applications, the answer is **no**.

Soft Deletes provide a way to mark records as deleted without physically removing them from the database.

This allows applications to:

- Recover deleted records
- Preserve historical data
- Maintain audit trails
- Improve data safety
- Reduce accidental data loss

---

# What Is A Soft Delete?

A soft delete marks a record as deleted rather than removing it.

Instead of:

```text
DELETE FROM users
```

The application updates the record:

```text
deleted_at = 2026-06-09 12:00:00
```

The record remains in the database.

---

# Hard Delete vs Soft Delete

## Hard Delete

```text
Record Exists
      │
      ▼
Delete
      │
      ▼
Record Removed Forever
```

The data no longer exists.

---

## Soft Delete

```text
Record Exists
      │
      ▼
Delete
      │
      ▼
Record Marked As Deleted
```

The data remains available for recovery.

---

# Why Soft Deletes Exist

Without soft deletes:

```text
Delete User
      │
      ▼
Data Permanently Lost
```

Problems:

```text
Accidental Deletion

Lost Business Data

No Recovery

Broken Audit Trails
```

Soft deletes provide a safety net.

---

# Soft Delete Philosophy

Deleting data should not always mean destroying data.

Instead:

```text
Hidden From Application
```

while remaining:

```text
Stored In Database
```

This approach is especially useful for business-critical systems.

---

# Soft Delete Lifecycle

A typical lifecycle:

```text
Create Record
      │
      ▼
Active Record
      │
      ▼
Soft Delete
      │
      ▼
Archived Record
      │
      ▼
Restore
      │
      ▼
Active Record
```

Records can move between active and deleted states.

---

# Database Structure

Soft deletes typically require a dedicated column.

Example:

```text
users
├── id
├── name
├── email
└── deleted_at
```

The `deleted_at` field indicates whether a record has been deleted.

---

# Active Records

An active record contains:

```text
deleted_at = NULL
```

Example:

```json
{
  "id": 1,
  "name": "John",
  "deleted_at": null
}
```

The record is considered active.

---

# Deleted Records

A deleted record contains a timestamp.

Example:

```json
{
  "id": 1,
  "name": "John",
  "deleted_at":
  "2026-06-09T12:00:00Z"
}
```

The record is considered deleted.

---

# How Soft Deletes Work

When deleting:

```text
DELETE
```

becomes:

```text
UPDATE
```

The framework updates:

```text
deleted_at
```

instead of removing the row.

---

# Query Behavior

Applications usually exclude soft deleted records automatically.

Example:

```text
SELECT Active Records
```

Result:

```text
deleted_at IS NULL
```

This behavior is often implemented through a global scope.

---

# Global Filtering

Flow:

```text
Query
  │
  ▼
Soft Delete Scope
  │
  ▼
Database
```

Deleted records remain hidden by default.

---

# Deleting A Record

Example:

```ts
await user.delete();
```

Internally:

```text
Set deleted_at
```

rather than:

```text
Remove Row
```

The record remains recoverable.

---

# Restoring Records

A soft deleted record can be restored.

Example:

```ts
await user.restore();
```

Flow:

```text
Deleted Record
      │
      ▼
Restore
      │
      ▼
Active Record
```

The record becomes visible again.

---

# Force Deleting

Sometimes permanent removal is necessary.

Example:

```ts
await user.forceDelete();
```

Flow:

```text
Database Row
      │
      ▼
Permanent Removal
```

After a force delete, recovery is no longer possible.

---

# Recovery Benefits

Soft deletes allow applications to recover from mistakes.

Example:

```text
Admin Deletes User
```

Later:

```text
User Restored
```

No data is lost.

---

# Audit Benefits

Deleted records remain available for auditing.

Examples:

```text
Who Was Deleted?

When Were They Deleted?

How Often Are Records Removed?
```

Historical visibility is preserved.

Example:

```text
Deleted Records Remain Stored
```

Benefits:

```text
Recovery

Auditing
```

Trade-off:

```text
Additional Storage Usage
```

# Archiving Strategies

Flow:

```text

      │
      ▼

      │

  ▼
```

```text
  ▼
```

Example:

# Common Mistakes

### Assuming Soft Deletes Remove Data

The record still exists in the database.

---

### Ignoring Storage Growth

Large numbers of deleted records can impact performance.

---

### Forgetting Restore Logic

Restoration should be part of the design process.

---

### Exposing Deleted Records Accidentally

Ensure APIs and queries respect soft delete filters.

---

### Using Soft Deletes For Everything

Not all tables require recoverable records.

Soft deletes provide flexibility while reducing the risk of accidental data loss.

# Key Takeaways

✔ Soft deletes preserve records instead of permanently removing them

✔ Deleted records are typically marked using a `deleted_at` column

✔ Active records have `deleted_at = NULL`

✔ Soft deleted records can be restored

✔ Force deletes permanently remove records

✔ Soft deletes improve recovery and auditing capabilities

✔ Global scopes often hide deleted records automatically

✔ Not every table requires soft delete functionality

Soft Deletes are a valuable feature of the Bejibun ORM because they provide safer data management workflows while preserving historical information and recovery capabilities.
