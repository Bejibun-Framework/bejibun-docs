`@bejibun/cors` manages Cross-Origin Resource Sharing. Headers are computed once at startup and
merged into every response automatically.

## Installation

```bash
bun ace install @bejibun/cors
```

This installs the package and drops `config/cors.ts`.

## Configuration

```ts config/cors.ts
const config: Record<string, any> = {
    allowedHeaders: "*",
    credentials: false,
    exposedHeaders: [],
    maxAge: 86400,
    methods: "*",
    origin: "*"
};

export default config;
```

| Option           | Description                        |
| ---------------- | ---------------------------------- |
| `origin`         | Allowed origin(s)                  |
| `methods`        | Allowed HTTP methods               |
| `allowedHeaders` | Headers the client may send        |
| `exposedHeaders` | Headers exposed to the client      |
| `credentials`    | Allow cookies / credentials        |
| `maxAge`         | Preflight cache duration (seconds) |

## How it applies

You normally don't touch CORS per request.
The configured headers are applied to every response built with the `Response` facade (both `send()` and `stream()`).

When returning a raw `Response`, merge the headers in manually:

<CodeGroup>
    ```ts Standalone
    import Cors from "@bejibun/cors";

    return Response.json(payload, {
        headers: {
            ...Cors.init
        }
    });
    ```

    ```ts Using Bejibun
    import CorsLoader from "@bejibun/core/loader/CorsLoader";
    import Cors from "@bejibun/cors";

    return Response.json(payload, {
        headers: {
            ...CorsLoader.cors
        }
    });
    ```
</CodeGroup>

<Warning>
    The defaults (`origin: "*"`, `allowedHeaders: "*"`) are permissive for getting started.
    In production, replace `*` with an explicit allowlist of trusted origins and headers,
    and only enable `credentials` when you actually need cookies or authorization headers across origins.
</Warning>
