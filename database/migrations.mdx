# Migrations

Migrations are version-controlled database schema definitions.

They allow developers to create, modify, and remove database structures using code rather than manually executing SQL statements.

Instead of treating the database as a separate system, migrations make the database schema part of your application's source code.

This provides:

- Consistency
- Reproducibility
- Collaboration
- Version control
- Safe deployments

Migrations are the foundation of database schema management in Bejibun.

---

# What Is A Migration?

A migration is a file that describes a database change.

Examples:

```text
Create Users Table

Add Email Column

Create Orders Table

Drop Legacy Table
```

Each migration represents a single, trackable schema change.

---

# Why Migrations Exist

Without migrations:

```text
Developer A
      │
      ▼
Manual Database Changes

Developer B
      │
      ▼
Different Database Structure
```

This often leads to:

```text
Inconsistent Environments

Deployment Failures

Lost Schema Changes

Difficult Collaboration
```

Migrations solve these problems by making schema changes explicit and repeatable.

---

# Migration Philosophy

Database structures should evolve in the same way application code evolves.

Instead of:

```text
Database Changes
     │
     ▼
Manual Process
```

Bejibun encourages:

```text
Database Changes
     │
     ▼
Migration Files
     │
     ▼
Version Control
```

The database becomes part of the application's history.

---

# Migration Lifecycle

A migration typically follows this process:

```text
Create Migration
       │
       ▼
Run Migration
       │
       ▼
Database Updated
```

If necessary:

```text
Rollback Migration
       │
       ▼
Previous State Restored
```

This allows schema changes to move both forward and backward safely.

---

# Creating A Migration

A migration begins as a dedicated file.

Example:

```text
create_users_table
```

Purpose:

```text
Create Database Structure

Modify Existing Tables

Remove Obsolete Structures
```

Each migration should have a clear responsibility.

---

# Migration Structure

A migration usually contains two methods.

Example:

```ts
export default class
CreateUsersTable {
  async up() {}

  async down() {}
}
```

Purpose:

```text
up()   → Apply Changes

down() → Revert Changes
```

This dual-direction design enables safe rollbacks.

---

# The Up Method

The `up()` method defines the schema changes that should be applied.

Example:

```text
Create Table

Add Columns

Create Indexes

Create Constraints
```

These operations move the database forward.

---

# The Down Method

The `down()` method reverses the changes made by `up()`.

Example:

```text
Drop Table

Remove Columns

Drop Indexes

Remove Constraints
```

This allows migrations to be safely rolled back.

Example:

```text

users.id
```

Benefits:

```text

Relationship Validation

Consistent Data
```

Foreign keys enforce valid relationships.

# Relationship Example

```text

  │
  └── Posts
```

```text

```

references:

```text

```

# Soft Deletes

Some applications preserve deleted records.

Additional column:

```text

```

Flow:

```text

      │
      ▼

```

# Running Migrations

Running migrations applies pending schema changes.

Flow:

```text
Migration Files
        │
        ▼
Migration Runner
        │
        ▼
Database Updated
```

The framework executes migrations in order.

---

# Migration Tracking

The framework tracks executed migrations.

Example:

```text
Migration History
```

Benefits:

```text
Prevent Duplicate Execution

Safe Deployments

Rollback Support
```

Each migration runs only once.

---

# Migration Order

Migrations execute chronologically.

Example:

```text
001_create_users

002_create_posts

003_add_email_to_users
```

Order matters because later migrations may depend on earlier structures.

---

# Rolling Back Migrations

Rollbacks reverse previously applied migrations.

Flow:

```text
Applied Migration
       │
       ▼
Rollback
       │
       ▼
Previous Schema
```

Useful during development and deployments.

---

# Migration Batches

Migrations are often grouped into batches.

Example:

```text
Batch 1
 ├── Migration A
 └── Migration B

Batch 2
 ├── Migration C
 └── Migration D
```

Rollback operations commonly target the most recent batch.

---

# Fresh Migrations

Development environments may rebuild the database completely.

Flow:

```text
Drop Database Structure
       │
       ▼
Run All Migrations
```

Useful during active development.

---

# Migration Safety

Important considerations:

```text

Downtime

Compatibility
```

# Backward Compatibility

Good migrations should minimize disruption.

Examples:

```text

Gradual Rollouts

Safe Defaults
```

Backward-compatible migrations reduce deployment risk.

# Best Practices

## One Responsibility Per Migration

Good:

```text
Create Users Table
```

Avoid:

```text
Create Users
Create Posts
Create Orders
Create Products
```

in a single migration.

---

## Always Implement Rollbacks

Every migration should be reversible whenever possible.

---

## Keep Migrations Small

Smaller migrations are easier to understand and maintain.

---

## Use Foreign Keys

Protect data integrity at the database level.

---

## Commit Migrations To Source Control

Treat migrations as application code.

---

# Common Mistakes

### Editing Old Migrations

Once migrations have been shared, create new migrations instead of modifying existing ones.

---

### Large Destructive Changes

Dropping columns or tables can permanently remove data.

---

### Missing Rollbacks

Always provide a path back when possible.

---

### Ignoring Indexes

Performance issues often originate from missing indexes.

---

### Combining Multiple Concerns

Keep migrations focused and easy to review.
