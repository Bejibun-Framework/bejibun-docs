# Models

Models are the primary way your application interacts with database records.

A model represents a table in your database and provides an object-oriented interface for creating, retrieving, updating, and deleting data.

Instead of working directly with raw SQL, models allow you to interact with data using familiar TypeScript classes.

Models are one of the most important building blocks of the Bejibun database layer.

---

# What Is A Model?

A model is a class that represents a database table.

Example:

```text
User Model
      │
      ▼
users Table
```

Every instance of the model represents a single database record.

Example:

```json
{
  "id": 1,
  "name": "John Doe",
  "email": "john@example.com"
}
```

This record becomes a model object that can be manipulated within your application.

---

# Why Models Exist

Without models:

```text
Application
     │
     ▼
Raw SQL
```

Every query would need to be written manually.

Example:

```sql
SELECT *
FROM users
WHERE id = 1;
```

As applications grow:

```text
Queries Become Repetitive

Business Rules Become Scattered

Maintenance Becomes Difficult
```

Models provide a centralized and expressive interface for working with data.

---

# Model Lifecycle

A typical model lifecycle looks like this:

```text
Database Record
        │
        ▼
Model Instance
        │
        ▼
Application Logic
        │
        ▼
Database Update
```

Models act as a bridge between the database and your application.

---

# Creating A Model

Models typically extend a base model class.

Example:

```ts
export default class User extends Model {
  //
}
```

This gives the model access to database functionality provided by the framework.

---

# Table Mapping

Each model is associated with a database table.

Example:

```text
User
```

maps to:

```text
users
```

Likewise:

```text
Post
```

maps to:

```text
posts
```

This convention allows the framework to locate the appropriate table automatically.

---

# Model Structure

Example:

```ts
export default class User extends Model {
  id!: number;

  name!: string;

  email!: string;
}
```

The model describes the shape of the underlying data.

---

# Model Instances

When data is retrieved from the database, it becomes a model instance.

Example:

```ts
const user =
  await User.find(1);
```

Result:

```ts
User
```

The returned object contains both data and model functionality.

---

# Retrieving Records

Models provide methods for retrieving data.

Example:

```ts
const user =
  await User.find(1);
```

Flow:

```text
Model
   │
   ▼
Query
   │
   ▼
Database
   │
   ▼
Result
```

The framework handles the underlying query automatically.

---

# Finding A Record

Retrieve a single record by its primary key.

Example:

```ts
const user =
  await User.find(1);
```

Result:

```json
{
  "id": 1,
  "name": "John Doe"
}
```

This is one of the most common model operations.

---

# Finding Or Failing

Some operations require a record to exist.

Example:

```ts
const user =
  await User.findOrFail(1);
```

Flow:

```text
Record Found
      │
      ▼
Return Model
```

or:

```text
Record Missing
      │
      ▼
Exception
```

This avoids repetitive existence checks.

---

# Retrieving Multiple Records

Example:

```ts
const users =
  await User.all();
```

Result:

```text
Collection<User>
```

Useful for listing resources.

---

# Creating Records

Models can create new database records.

Example:

```ts
const user =
  await User.create({
    name: "John Doe",
    email: "john@example.com"
  });
```

Flow:

```text
Model Data
     │
     ▼
Insert Query
     │
     ▼
Database
```

A new record is stored and returned.

---

# Updating Records

Existing records can be modified.

Example:

```ts
user.name =
  "Jane Doe";

await user.save();
```

Flow:

```text
Model
   │
   ▼
Update Query
   │
   ▼
Database
```

Changes are persisted automatically.

---

# Deleting Records

Models can remove records.

Example:

```ts
await user.delete();
```

Flow:

```text
Model
   │
   ▼
Delete Query
   │
   ▼
Database
```

The corresponding database row is removed.

---

# Mass Assignment

Models may support creating records from structured input.

Example:

```ts
await User.create({
  name: "John",
  email: "john@example.com"
});
```

Mass assignment simplifies record creation.

---

# Model Attributes

Attributes represent database columns.

Example:

```ts
user.name

user.email

user.createdAt
```

Each attribute maps to a stored value.

---

# Attribute Casting

Database values can be automatically converted into application-friendly types.

Database:

```json
{
  "is_active": 1
}
```

Model:

```ts
{
  isActive: true
}
```

Casting improves consistency throughout the application.

---

# Default Values

Models may define default values for attributes.

Example:

```text
status = active
```

This helps reduce repetitive initialization logic.

---

# Hidden Attributes

Some fields should never be exposed publicly.

Examples:

```text
Password

Tokens

Secrets
```

Hidden attributes are excluded during serialization.

---

# Computed Attributes

Models can expose calculated values.

Example:

```ts
fullName
```

Generated from:

```ts
firstName

lastName
```

Computed properties improve usability without storing redundant data.

---

# Model Serialization

Models can be converted into JSON.

Example:

```ts
user.toJSON();
```

Result:

```json
{
  "id": 1,
  "name": "John"
}
```

Serialization is commonly used when returning API responses.

---

# Querying Through Models

Models often expose query-building capabilities.

Example:

```ts
User.query()
```

Flow:

```text
Model
   │
   ▼
Query Builder
   │
   ▼
Database
```

This provides a fluent way to retrieve data.

---

# Filtering Records

Example:

```ts
User.query()
  .where("active", true);
```

Result:

```text
Active Users Only
```

Filtering reduces unnecessary data retrieval.

---

# Ordering Records

Example:

```ts
User.query()
  .orderBy("name");
```

Result:

```text
Alphabetically Sorted Users
```

Ordering controls result presentation.

---

# Limiting Results

Example:

```ts
User.query()
  .limit(10);
```

Result:

```text
Maximum 10 Records
```

Useful for pagination and performance.

---

# Model Relationships

Models frequently relate to one another.

Example:

```text
User
 │
 └── Posts
```

Relationships allow applications to navigate connected data naturally.

---

# One-To-One Relationships

Example:

```text
User
 │
 └── Profile
```

Each user owns exactly one profile.

---

# One-To-Many Relationships

Example:

```text
User
 │
 ├── Post
 ├── Post
 └── Post
```

A single user owns many posts.

---

# Many-To-Many Relationships

Example:

```text
User
 │
 └── Roles
```

Users may have multiple roles and roles may belong to multiple users.

---

# Relationship Loading

Relationships can be loaded alongside a model.

Example:

```text
User
 └── Posts
```

Benefits:

```text
Fewer Queries

Better Performance

Cleaner Code
```

---

# Model Events

Applications may react to model lifecycle events.

Examples:

```text
Creating

Created

Updating

Updated

Deleting

Deleted
```

Events allow custom behavior during model operations.

---

# Soft Deletes

Sometimes records should not be permanently removed.

Instead:

```text
Record Marked As Deleted
```

Benefits:

```text
Recovery

Auditing

Historical Tracking
```

Soft deletes provide additional safety.

---

# Model Factories

Factories generate model instances for testing and seeding.

Example:

```text
Generate User
```

Benefits:

```text
Testing

Development

Seed Data
```

Factories reduce repetitive setup code.

---

# Model Scopes

Scopes provide reusable query constraints.

Example:

```ts
User.active()
```

Instead of:

```ts
User.query()
  .where("active", true);
```

Scopes improve readability and reuse.

---

# Models And Controllers

Controllers should use models indirectly whenever possible.

Example:

```text
Controller
     │
     ▼
Service
     │
     ▼
Model
```

This separation keeps controllers focused on request handling.

---

# Models And Services

Services frequently coordinate model operations.

Example:

```text
UserService
      │
      ▼
User Model
```

Business logic belongs in services rather than models or controllers.

---

# Models And Resources

Models represent data.

Resources represent presentation.

```text
Model
   │
   ▼
Resource
   │
   ▼
API Response
```

Each layer serves a different purpose.

---

# Best Practices

## Keep Models Focused

Models should represent data and relationships.

---

## Use Services For Business Logic

Avoid placing large workflows directly inside models.

---

## Hide Sensitive Fields

Protect passwords, tokens, and private information.

---

## Use Relationships

Avoid manually joining related data everywhere.

---

## Prefer Query Builder Methods

Use expressive model queries instead of raw SQL whenever possible.

---

# Common Mistakes

### Fat Models

Avoid placing all application logic inside models.

---

### Exposing Sensitive Data

Never return passwords or internal secrets.

---

### Duplicate Queries

Use scopes and reusable query patterns.

---

### Ignoring Relationships

Relationships simplify data access significantly.

---

### Excessive Raw SQL

Use framework abstractions when appropriate.
