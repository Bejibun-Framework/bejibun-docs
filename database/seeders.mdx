# Seeders

Seeders are classes that populate the database with predefined data.

They allow applications to automatically insert records without requiring manual database operations.

Seeders are commonly used for:

- Development data
- Test data
- Initial application setup
- Default configuration
- Roles and permissions
- Reference data

Instead of manually creating records every time a database is created, seeders automate the process.

---

# What Is A Seeder?

A seeder is a class responsible for inserting data into the database.

Example:

```text
User Seeder

Role Seeder

Permission Seeder

Settings Seeder
```

Each seeder typically focuses on a specific set of records.

---

# Why Seeders Exist

Without seeders:

```text
Create Database
       │
       ▼
Manually Insert Data
       │
       ▼
Repeat For Every Environment
```

This becomes:

```text
Slow

Error-Prone

Inconsistent
```

Seeders automate the process.

---

# Seeder Philosophy

Database structure and database content serve different purposes.

```text
Migration
     │
     ▼
Database Structure
```

```text
Seeder
     │
     ▼
Database Content
```

Migrations define tables.

Seeders populate those tables.

---

# Seeder Lifecycle

A typical seeding workflow:

```text
Run Migrations
       │
       ▼
Create Tables
       │
       ▼
Run Seeders
       │
       ▼
Insert Records
```

Seeders usually run after migrations.

---

# Creating A Seeder

A seeder begins as a dedicated class.

Example:

```ts
export default class UserSeeder {
  async run() {}
}
```

The `run()` method contains the data insertion logic.

# Inserting Records

A seeder may create records directly through models.

Example:

```ts
await User.create({
  name: "John Doe",
  email: "john@example.com"
});
```

This inserts a new database record.

# Database Seeder

Applications commonly define a root seeder.

Example:

```text
DatabaseSeeder
```

Purpose:

```text
Execute Other Seeders
```

Flow:

```text
DatabaseSeeder
     │
     ├── UserSeeder
     ├── RoleSeeder
     └── SettingsSeeder
```

This becomes the application's entry point for seeding.

# Best Practices

## Keep Seeders Focused

One seeder should handle one responsibility.

Good:

```text
RoleSeeder
```

Avoid:

```text
EverythingSeeder
```

---

## Use A Root Seeder

Create a central entry point that executes all required seeders.

---

## Seed Only Necessary Production Data

Avoid filling production databases with sample records.

---

## Make Seeders Re-Runnable

Prevent duplicate records whenever possible.

---

## Use Factories For Large Datasets

Factories simplify generating large amounts of data.

---

# Common Mistakes

### Mixing Migrations And Seeders

Migrations define structure.

Seeders define content.

Keep them separate.

---

### Hardcoding Environment-Specific Data

Seeders should remain reusable.

---

### Creating Duplicate Records

Always consider repeat execution.

---

### Seeding Excessive Production Data

Production databases should contain only necessary records.

---

### Ignoring Relationships

Related data should be seeded consistently.
