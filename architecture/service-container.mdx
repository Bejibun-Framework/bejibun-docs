# Service Container

The Service Container is the foundation of dependency management in Bejibun.

It is responsible for creating, resolving, and managing objects throughout your application.

Rather than manually instantiating classes, you can allow the container to automatically provide the dependencies your application requires.

## Why Use a Service Container?

As applications grow, classes often depend on other classes.

Without a container:

```ts id="j83l2p"
const logger = new Logger();
const repository = new UserRepository(logger);
const service = new UserService(repository);
const controller = new UserController(service);
```

Managing dependencies manually quickly becomes difficult and error-prone.

The Service Container solves this by automatically resolving dependencies.

```ts id="x2v0w7"
const controller = container.make(UserController);
```

The container creates all required dependencies automatically.

---

## What Problems Does It Solve?

The Service Container provides:

* Automatic dependency resolution
* Dependency Injection support
* Centralized service registration
* Loose coupling between classes
* Easier testing and mocking
* Better application architecture

Without a container, classes often become tightly coupled to their implementations.

With a container, classes depend on abstractions instead.

---

## How It Works

Consider the following controller:

```ts id="5j5w7n"
export class UserController {
  constructor(
    private userService: UserService
  ) {}
}
```

And the service:

```ts id="z98bq2"
export class UserService {
  constructor(
    private repository: UserRepository
  ) {}
}
```

When the controller is resolved, the container automatically performs:

```text id="8s9o6z"
UserController
       │
       ▼
 UserService
       │
       ▼
UserRepository
```

Each dependency is constructed and injected automatically.

---

## Resolving Services

Services can be resolved directly from the container.

```ts id="m9yx8e"
const service = container.make(UserService);
```

The container will:

1. Inspect constructor dependencies
2. Resolve required services
3. Create instances
4. Return the final object

```text id="r8m2zv"
Container
    │
    ▼
Resolve Dependencies
    │
    ▼
Create Instance
    │
    ▼
Return Object
```

This process is called **resolution**.

---

## Constructor Injection

The most common way to receive dependencies is through constructor injection.

```ts id="s0v2fq"
export class UserService {
  constructor(
    private logger: Logger
  ) {}
}
```

When the service is resolved, the logger is automatically provided.

```ts id="q0zj4r"
const service = container.make(UserService);
```

Constructor injection makes dependencies explicit and easy to understand.

### Benefits

* Clear dependencies
* Better testing
* Improved maintainability
* Reduced coupling

---

## Dependency Graph

The container can resolve entire dependency trees.

```text id="zrqzdh"
UserController
       │
       ▼
UserService
       │
       ▼
UserRepository
       │
       ▼
DatabaseConnection
```

When the controller is requested, every dependency beneath it is resolved automatically.

This eliminates the need for manual object construction.

---

## Registering Services

While many classes can be resolved automatically, some services must be registered explicitly.

This is typically done inside a Service Provider.

```ts id="0tzv3y"
export class AppServiceProvider {
  register() {
    container.bind(
      UserRepository,
      DatabaseUserRepository
    );
  }
}
```

This tells the container which implementation should be used when resolving a dependency.

---

## Bindings

A binding maps an abstraction to a concrete implementation.

```ts id="o7iy1n"
container.bind(
  MailService,
  SmtpMailService
);
```

Now whenever `MailService` is requested, the container returns an instance of `SmtpMailService`.

```text id="m4dt3k"
MailService
      │
      ▼
SmtpMailService
```

This allows implementations to be replaced without modifying application code.

---

## Singleton Services

Some services should only be created once during the application's lifetime.

Examples include:

* Database connections
* Cache managers
* Event dispatchers
* Configuration services

A singleton binding ensures only one instance exists.

```ts id="t8wyj0"
container.singleton(
  Logger,
  Logger
);
```

The first resolution creates the instance.

Subsequent resolutions reuse the same object.

```text id="v9fj7m"
Request #1
     │
     ▼
 Logger Instance
     ▲
     │
Request #2
```

---

## Instance Bindings

You may register an already-created object.

```ts id="5w1s6l"
const logger = new Logger();

container.instance(
  Logger,
  logger
);
```

The container will always return the registered instance.

This is useful for configuration-driven services.

---

## Interface-Based Design

Applications should depend on abstractions rather than implementations.

```ts id="g0b6qg"
export abstract class MailService {
  abstract send(): Promise<void>;
}
```

Implementation:

```ts id="0yjxjp"
export class SmtpMailService
  extends MailService {

  async send() {
    //
  }
}
```

Binding:

```ts id="3nfh1v"
container.bind(
  MailService,
  SmtpMailService
);
```

Usage:

```ts id="km5ixh"
export class UserService {
  constructor(
    private mail: MailService
  ) {}
}
```

This makes implementations easy to swap and mock.

---

## Container and Controllers

Controllers are automatically resolved through the container.

```ts id="v9bq0w"
router.get(
  "/users",
  [UserController, "index"]
);
```

When the route executes:

```text id="n5o6eq"
Route
   │
   ▼
Container
   │
   ▼
Controller
```

Dependencies are injected automatically before the controller method runs.

---

## Container and Middleware

Middleware may also receive dependencies from the container.

```ts id="g5r6s0"
export class AuthMiddleware {
  constructor(
    private auth: AuthService
  ) {}
}
```

The framework resolves middleware instances the same way it resolves controllers.

---

## Testing with the Container

One major advantage of a service container is improved testability.

Production:

```ts id="zv7xgn"
container.bind(
  MailService,
  SmtpMailService
);
```

Testing:

```ts id="5lkg6z"
container.bind(
  MailService,
  FakeMailService
);
```

No application code needs to change.

This allows dependencies to be replaced with test doubles during automated testing.

## Best Practices

  <Card title="Prefer Constructor Injection">

  </Card>

  <Card title="Use Singletons Carefully">
    Reserve singleton bindings for truly shared services.

  <Card title="Keep Providers Focused">

## Common Mistakes

### Creating Dependencies Manually

```ts id="xhnfmu"
export class UserService {
  private logger = new Logger();
}
```

This tightly couples the service to a specific implementation.

### Preferred Approach

```ts id="93i27u"
export class UserService {
  constructor(
    private logger: Logger
  ) {}
}
```

The container remains responsible for dependency creation.

## Summary

The Service Container is responsible for creating and managing application dependencies.

It provides:

* Automatic dependency resolution
* Constructor injection
* Service bindings
* Singleton management
* Interface-based architecture
* Improved testability

The container powers many core Bejibun features, including controllers, middleware, services, commands, and package integrations.
