# Service Providers

Service Providers are the central place for configuring and bootstrapping your application.

Almost every framework feature is registered through a Service Provider, including database connections, cache drivers, event listeners, custom services, and package integrations.

If the Service Container is the heart of dependency management, Service Providers are the mechanism that teaches the container how your application should be configured.

## What Are Service Providers?

A Service Provider is a class responsible for:

- Registering services in the Service Container
- Configuring framework components
- Loading application resources
- Bootstrapping functionality
- Initializing package integrations

A typical provider looks like this:

```ts
export class AppServiceProvider {
  register() {
    //
  }

  boot() {
    //
  }
}
```

Most application initialization should happen inside Service Providers.

---

## Why Service Providers Exist

Without Service Providers, application startup code often becomes scattered throughout the project.

```ts
container.bind(Cache, RedisCache);
container.bind(Mailer, SmtpMailer);
container.bind(Storage, LocalStorage);

// More setup...
```

As the application grows, configuration becomes difficult to manage.

Service Providers organize startup logic into dedicated classes.

```ts
export class CacheServiceProvider {
  register() {
    //
  }
}
```

This makes application bootstrapping predictable and maintainable.

---

## The Application Bootstrap Process

When Bejibun starts, it loads and executes registered Service Providers.

```text
Application Start
        │
        ▼
Load Configuration
        │
        ▼
Register Providers
        │
        ▼
Boot Providers
        │
        ▼
Application Ready
```

This process occurs before requests are handled.

---

## Registering Providers

Providers are typically registered in the application's configuration.

```ts
export default {
  providers: [
    AppServiceProvider,
    DatabaseServiceProvider,
    CacheServiceProvider,
  ],
};
```

When the application starts, Bejibun loads each provider in order.

---

## The Register Method

The `register()` method is used to register services with the Service Container.

```ts
export class AppServiceProvider {
  register() {
    container.bind(
      UserRepository,
      DatabaseUserRepository
    );
  }
}
```

The purpose of `register()` is to tell the container what services exist and how they should be resolved.

### Common Tasks

Inside `register()` you may:

- Register bindings
- Register singletons
- Register drivers
- Register package services

Example:

```ts
register() {
  container.singleton(
    Logger,
    Logger
  );
}
```

Avoid executing application logic inside `register()`.

---

## The Boot Method

The `boot()` method executes after all providers have been registered.

```ts
export class AppServiceProvider {
  boot() {
    //
  }
}
```

At this stage:

- All services have been registered
- All dependencies are available
- Providers may interact with each other

### Common Tasks

Inside `boot()` you may:

- Register event listeners
- Configure middleware
- Register routes
- Extend framework components
- Execute startup logic

Example:

```ts
boot() {
  events.listen(
    UserCreated,
    SendWelcomeEmail
  );
}
```

---

## Register vs Boot

Understanding the difference between these methods is important.

### Register

Used for service registration.

```ts
register() {
  container.bind(
    MailService,
    SmtpMailService
  );
}
```

### Boot

Used for service usage and initialization.

```ts
boot() {
  const mail = container.make(MailService);

  //
}
```

### Rule of Thumb

| Method | Purpose |
|----------|----------|
| `register()` | Register services |
| `boot()` | Use services |

## Creating Custom Providers

As applications grow, separating functionality into dedicated providers improves organization.

### Example

```ts
export class PaymentServiceProvider {
  register() {
    container.bind(
      PaymentGateway,
      StripeGateway
    );
  }
}
```

Register the provider:

```ts
export default {
  providers: [
    PaymentServiceProvider,
  ],
};
```

The service becomes available throughout the application.

---

## Organizing Providers

Large applications often use multiple providers.

```text
app/
├── Providers/
│   ├── AppServiceProvider.ts
│   ├── AuthServiceProvider.ts
│   ├── EventServiceProvider.ts
│   ├── CacheServiceProvider.ts
│   └── RouteServiceProvider.ts
```

Each provider should focus on a specific responsibility.

---

## Service Provider Example

A complete provider might look like this:

```ts
export class AppServiceProvider {
  register() {
    container.singleton(
      Logger,
      Logger
    );

    container.bind(
      MailService,
      SmtpMailService
    );
  }

  boot() {
    events.listen(
      UserCreated,
      SendWelcomeEmail
    );
  }
}
```

During startup:

```text
Register Phase
    │
    ├── Logger
    └── MailService

Boot Phase
    │
    └── Event Listeners
```

## Package Service Providers

Framework packages commonly expose Service Providers.

For example:

```ts
import { RedisServiceProvider } from "@bejibun/redis";
```

Registering the provider enables package functionality.

```ts
export default {
  providers: [
    RedisServiceProvider,
  ],
};
```

This keeps package setup simple and consistent.

---

## Deferred Loading

Some providers may only be loaded when their services are needed.

```text
Application Start
        │
        ▼
Provider Registered
        │
        ▼
Service Requested
        │
        ▼
Provider Loaded
```

Deferred loading can improve application startup performance.

> The availability of deferred providers depends on the specific implementation and package support.

---

## Provider Dependencies

Providers may depend on services registered by other providers.

For example:

```text
DatabaseProvider
        │
        ▼
UserRepositoryProvider
        │
        ▼
AuthProvider
```

Because providers are loaded sequentially, registration order can be important.

When possible, keep providers independent and focused.

---

## Common Use Cases

Service Providers are commonly used for:

- Service Container bindings
- Database setup
- Cache configuration
- Event registration
- Storage drivers
- Authentication services
- Package initialization
- Framework extensions

If something needs to be configured during startup, a Service Provider is usually the right place for it.

## Common Mistakes

### Using Services During Registration

```ts
register() {
  const mail = container.make(MailService);

  // Avoid
}
```

At registration time, not all services may be available.

Use `boot()` instead.

### Mixing Responsibilities

```ts
export class AppServiceProvider {
  register() {
    // Database
    // Cache
    // Auth
    // Events
    // Storage
    // Everything else...
  }
}
```

Large providers become difficult to maintain.

Split functionality into dedicated providers.

Service Providers are responsible for bootstrapping and configuring your application.

- Organize startup logic

Nearly every major framework feature is powered by Service Providers, making them one of the most important building blocks of a Bejibun application.
