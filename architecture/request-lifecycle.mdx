# Request Lifecycle

Understanding the request lifecycle helps you know exactly what happens when a client sends a request to your application.

Every request in Bejibun follows a predictable execution flow, allowing you to intercept, transform, validate, and respond to requests in a consistent manner.

## Overview

A typical request passes through several layers before a response is returned.

```text
Client Request
       │
       ▼
   HTTP Server
       │
       ▼
 Application Boot
       │
       ▼
 Service Providers
       │
       ▼
   Middleware
       │
       ▼
     Router
       │
       ▼
  Controller
       │
       ▼
 Services / Models
       │
       ▼
   Response
       │
       ▼
Client
```

Each layer has a specific responsibility within the application.

---

## Step 1: Incoming Request

The lifecycle begins when a client sends an HTTP request.

```http
GET /users HTTP/1.1
Host: api.example.com
```

Requests may originate from:

* Web browsers
* Mobile applications
* Server-to-server communication
* Third-party integrations
* API testing tools

The Bun runtime receives the request and forwards it to the Bejibun application.

---

## Step 2: Application Bootstrapping

Before the request is processed, Bejibun initializes the application.

This includes:

* Loading configuration
* Loading environment variables
* Initializing the service container
* Registering service providers
* Preparing application services

```ts
import { Application } from "@bejibun/core";

const app = new Application();
```

The bootstrap process ensures all required services are available before handling requests.

---

## Step 3: Service Provider Registration

Service Providers are responsible for registering services within the container.

During application startup, Bejibun loads and registers all configured providers.

```ts
export class AppServiceProvider {
  register() {
    //
  }

  boot() {
    //
  }
}
```

Providers may register:

* Database connections
* Cache drivers
* Storage services
* Event listeners
* Custom application services

This process usually happens once during application initialization.

---

## Step 4: Route Resolution

After bootstrapping, the router attempts to match the incoming request to a registered route.

```ts
router.get("/users", [UserController, "index"]);
```

For a request such as:

```http
GET /users
```

The router locates the corresponding route definition and determines which controller action should handle the request.

If no route matches, Bejibun returns a `404 Not Found` response.

---

## Step 5: Middleware Execution

Before reaching the controller, the request passes through middleware.

Middleware can inspect, modify, or reject requests.

```ts
router.get("/users", [UserController, "index"])
  .middleware(["auth"]);
```

Common middleware responsibilities include:

* Authentication
* Authorization
* Rate limiting
* Request logging
* Request transformation
* Security checks

### Example

```ts
export class AuthMiddleware {
  async handle(req, next) {
    if (!req.user()) {
      throw new UnauthorizedException();
    }

    return next();
  }
}
```

Middleware executes in the order it is registered.

```text
Request
   │
   ▼
Auth Middleware
   │
   ▼
Rate Limit Middleware
   │
   ▼
Controller
```

---

## Step 6: Dependency Resolution

Before invoking a controller, Bejibun resolves dependencies through the Service Container.

```ts
export class UserController {
  constructor(
    private userService: UserService
  ) {}
}
```

The container automatically creates and injects required dependencies.

```text
Controller
    │
    ▼
Service Container
    │
    ▼
Resolved Services
```

This allows controllers to remain focused on handling requests rather than creating dependencies.

---

## Step 7: Controller Execution

Once all middleware and dependencies have been processed, the controller action is executed.

```ts
export class UserController {
  async index(req, res) {
    return res.json(
      await this.userService.all()
    );
  }
}
```

Controllers typically:

* Receive request data
* Call services
* Interact with models
* Return responses

Controllers should avoid containing complex business logic.

---

## Step 8: Business Logic and Data Access

Most application logic is handled by services and models.

```ts
const users = await this.userService.all();
```

Services may:

* Validate business rules
* Coordinate multiple models
* Trigger events
* Execute background jobs
* Integrate external APIs

Models interact directly with the database.

```ts
const users = await User.query().get();
```

This layer contains the core behavior of your application.

---

## Step 9: Response Generation

After processing is complete, a response is generated.

```ts
return res.json(users);
```

Bejibun supports multiple response types:

```ts
return res.json(data);
return res.text("Hello");
return res.html(html);
return res.redirect("/dashboard");
```

The framework converts the response into a valid HTTP response before sending it back to the client.

---

## Step 10: Response Middleware

Certain middleware may execute additional logic after the controller has returned a response.

Examples include:

* Logging response data
* Measuring execution time
* Adding headers
* Tracking analytics

```text
Controller
    │
    ▼
Response Middleware
    │
    ▼
Client
```

This allows cross-cutting concerns to remain outside application logic.

---

## Exception Handling

Exceptions can occur at any stage of the request lifecycle.

```ts
throw new NotFoundException();
```

Bejibun catches uncaught exceptions and converts them into HTTP responses.

```json
{
  "message": "Resource not found"
}
```

Depending on configuration, exception handling may include:

* Structured JSON responses
* Detailed development errors
* Stack traces
* Logging
* Error reporting integrations

---

## Lifecycle Example

Consider the following request:

```http
GET /users
```

Execution flow:

```text
1. Request arrives
2. Application boots
3. Providers are loaded
4. Route is matched
5. Middleware executes
6. Dependencies are resolved
7. Controller runs
8. Models query database
9. Response is generated
10. Response sent to client
```

This predictable flow makes applications easier to debug and maintain.

## Summary

Every request in Bejibun follows a structured lifecycle:

1. Request arrives
2. Application initializes
3. Routes are resolved
4. Middleware executes
5. Dependencies are injected
6. Controllers run
7. Services and models perform work
8. Responses are generated
9. Exceptions are handled
10. The response is returned to the client

Understanding this lifecycle will help you build middleware, services, controllers, and packages that integrate naturally with the framework.
