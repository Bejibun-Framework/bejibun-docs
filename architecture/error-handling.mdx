# Error Handling

Errors are an inevitable part of software development.

Network failures, invalid input, missing resources, database outages, and unexpected bugs can all cause application failures. Bejibun provides a centralized error handling system that allows applications to gracefully handle these situations and return meaningful responses.

A consistent error handling strategy improves application reliability, debugging, and user experience.

## Why Error Handling Matters

Without proper error handling, exceptions can cause applications to crash or expose sensitive information.

Consider the following example:

```ts
const user = await User.find(id);

return user.name;
```

If the user does not exist:

```ts
TypeError: Cannot read properties of null
```

This creates an unhelpful experience for both developers and users.

Instead, errors should be handled intentionally.

```ts
if (!user) {
  throw new NotFoundException(
    "User not found"
  );
}
```

The framework can then generate an appropriate response.

---

## What Is an Exception?

An exception is an object that represents an error condition.

```ts
throw new Error("Something went wrong");
```

When an exception is thrown, normal application execution stops and control is passed to the exception handler.

```text
Application
      │
      ▼
 Exception Thrown
      │
      ▼
Exception Handler
      │
      ▼
HTTP Response
```

This allows errors to be handled consistently throughout the application.

---

## The Exception Lifecycle

When an exception occurs, Bejibun follows a predictable process.

```text
Request
   │
   ▼
Controller
   │
   ▼
Exception Thrown
   │
   ▼
Exception Handler
   │
   ▼
Response Generated
   │
   ▼
Client
```

The framework intercepts uncaught exceptions and converts them into HTTP responses.

## Throwing Exceptions

Exceptions may be thrown whenever an operation cannot be completed successfully.

```ts
throw new Error("Invalid operation");
```

More specific exceptions are preferred.

```ts
throw new NotFoundException(
  "User not found"
);
```

Specific exceptions communicate intent more clearly than generic errors.

---

## Common Exception Types

Applications commonly encounter the following exception categories.

### Not Found

```ts
throw new NotFoundException(
  "Post not found"
);
```

Used when a requested resource does not exist.

---

### Validation Errors

```ts
throw new ValidationException({
  email: ["The email field is required."],
});
```

Used when incoming data fails validation.

---

### Authentication Errors

```ts
throw new UnauthorizedException();
```

Used when a user is not authenticated.

---

### Authorization Errors

```ts
throw new ForbiddenException();
```

Used when a user lacks permission to perform an action.

---

### Server Errors

```ts
throw new Error(
  "Unexpected database failure"
);
```

Used for unexpected application failures.

---

## Automatic HTTP Responses

The framework automatically converts exceptions into HTTP responses.

Example:

```ts
throw new NotFoundException(
  "User not found"
);
```

Response:

```json
{
  "message": "User not found"
}
```

Status code:

```http
404 Not Found
```

This behavior keeps controllers clean and consistent.

---

## Handling Exceptions Manually

Sometimes exceptions should be handled directly.

```ts
try {
  await paymentService.charge();
} catch (error) {
  logger.error(error);

  return res.json({
    message: "Payment failed",
  }, 500);
}
```

Manual handling is useful when:

- Additional logging is required
- Alternative behavior is needed
- Recovery is possible

---

## Try-Catch Blocks

A `try-catch` block allows exceptions to be intercepted locally.

```ts
try {
  await processOrder();
} catch (error) {
  logger.error(error);
}
```

Execution flow:

```text
Try Block
    │
    ▼
 Exception?
    │
 ┌──┴──┐
 │ Yes │
 └──┬──┘
    ▼
 Catch Block
```

Use local exception handling only when necessary.

---

## Global Exception Handling

Most exceptions should be handled centrally.

```text
Application
      │
      ▼
Exception Handler
      │
      ▼
Response
```

Centralized handling provides:

- Consistent responses
- Unified logging
- Reduced duplication
- Easier maintenance

---

## Custom Exceptions

Applications may define custom exception types.

```ts
export class InsufficientBalanceException
  extends Error {

  constructor() {
    super("Insufficient balance");
  }
}
```

Usage:

```ts
throw new InsufficientBalanceException();
```

Custom exceptions make business rules easier to understand and manage.

---

## Domain-Specific Exceptions

Business logic often benefits from dedicated exception classes.

```ts
export class OrderAlreadyPaidException
  extends Error {}
```

```ts
if (order.isPaid()) {
  throw new OrderAlreadyPaidException();
}
```

This communicates intent more clearly than generic errors.

---

## Validation Exceptions

Validation failures are among the most common exceptions in web applications.

```ts
throw new ValidationException({
  email: [
    "The email field is required.",
  ],
});
```

Response:

```json
{
  "message": "Validation failed",
  "errors": {
    "email": [
      "The email field is required."
    ]
  }
}
```

Structured validation errors improve API usability.

---

## Reporting Exceptions

Not every exception should be displayed to users.

Instead, exceptions should be reported through logging or monitoring systems.

```ts
logger.error(error);
```

Reporting helps developers:

- Detect failures
- Investigate incidents
- Monitor production systems

---

## Logging Exceptions

Exceptions should typically be logged before being reported or rethrown.

```ts
try {
  await service.execute();
} catch (error) {
  logger.error("Service failed", {
    error,
  });

  throw error;
}
```

This creates an audit trail for troubleshooting.

---

## Error Responses

Production applications should provide clear but safe error responses.

Good:

```json
{
  "message": "Resource not found"
}
```

Bad:

```json
{
  "message": "SQL Error",
  "query": "...",
  "stack": "..."
}
```

Internal implementation details should never be exposed to clients.

---

## Development vs Production

Error handling behavior often differs by environment.

### Development

```json
{
  "message": "Database connection failed",
  "stack": "..."
}
```

### Production

```json
{
  "message": "Internal Server Error"
}
```

Detailed debugging information is useful during development but should not be exposed in production.

---

## HTTP Status Codes

Exceptions are commonly mapped to HTTP status codes.

| Exception | Status Code |
|------------|------------|
| Validation Exception | 422 |
| Unauthorized Exception | 401 |
| Forbidden Exception | 403 |
| Not Found Exception | 404 |
| Generic Error | 500 |

This provides predictable API behavior.

## Common Mistakes

### Swallowing Exceptions

```ts
try {
  await process();
} catch {
  //
}
```

Ignoring exceptions makes debugging extremely difficult.

---

### Exposing Internal Errors

```ts
return res.json({
  error,
});
```

Raw exceptions may contain sensitive information.

---

### Using Generic Errors Everywhere

```ts
throw new Error("Failed");
```

Prefer domain-specific exceptions whenever possible.

```ts
throw new UserNotFoundException();
```

A robust error handling strategy ensures that failures are handled gracefully and consistently across your entire application.
