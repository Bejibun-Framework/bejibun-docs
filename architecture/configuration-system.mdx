# Configuration System

Configuration is the process of defining how your application behaves in different environments.

Bejibun provides a centralized configuration system that allows applications to manage settings consistently without hardcoding values throughout the codebase.

The configuration system is responsible for:

- Application settings
- Database connections
- Cache configuration
- Storage drivers
- Redis connections
- Mail services
- WebSocket servers
- Framework behavior

A well-designed configuration system makes applications easier to maintain, deploy, and scale.

---

# Why Configuration Matters

Without centralized configuration, application settings often become scattered throughout the codebase.

Example:

```ts
const host = "localhost";
const port = 5432;
```

Later:

```ts
const redisHost = "localhost";
```

And elsewhere:

```ts
const cacheDriver = "memory";
```

As applications grow, managing these values becomes increasingly difficult.

The Configuration System solves this problem by centralizing application settings.

---

# Configuration Philosophy

Bejibun follows a simple principle:

```text
Environment Variables
          │
          ▼
Configuration Files
          │
          ▼
Application Services
          │
          ▼
Application Code
```

Application code should consume configuration rather than directly reading environment variables.

This creates a clean separation between infrastructure and business logic.

# Configuration Directory

Application configuration files are stored in:

```text
config/
```

Example:

```text
config/
├── app.ts
├── database.ts
├── cache.ts
├── redis.ts
├── storage.ts
├── mail.ts
└── websocket.ts
```

Each file is responsible for a specific area of configuration.

# Application Configuration

The application configuration file contains core framework settings.

Example:

```text
config/app.ts
```

Example structure:

```ts
export default {
  name: "Bejibun",
  env: "development",
  debug: true
};
```

Typical settings include:

- Application name
- Environment
- Debug mode
- Timezone
- Locale

---

# Environment Variables

Configuration values commonly originate from environment variables.

Example:

```env
APP_NAME=Bejibun
APP_ENV=production
APP_DEBUG=false
```

Configuration file:

```ts
export default {
  name: Env.get("APP_NAME"),
  env: Env.get("APP_ENV"),
  debug: Env.get("APP_DEBUG")
};
```

This allows behavior to change without modifying source code.

---

# Accessing Configuration

Configuration values can be accessed throughout the application.

Example:

```ts
Config.get("app.name");
```

Result:

```text
Bejibun
```

Configuration acts as a centralized source of truth.

---

# Nested Configuration

Configuration keys are commonly organized using dot notation.

Example:

```ts
Config.get(
  "database.connection"
);
```

or:

```ts
Config.get(
  "cache.driver"
);
```

This structure keeps configuration organized and predictable.

---

# Providing Default Values

Configuration lookups may provide fallback values.

Example:

```ts
Config.get(
  "cache.ttl",
  300
);
```

If the value is missing:

```text
300
```

is returned.

This helps prevent runtime errors.

---

# Database Configuration

Database settings typically live inside:

```text
config/database.ts
```

Example:

```ts
export default {
  connection: Env.get(
    "DB_CONNECTION"
  ),

  host: Env.get(
    "DB_HOST"
  ),

  port: Env.get(
    "DB_PORT"
  )
};
```

This separates database infrastructure from application code.

---

# Multiple Connections

Applications may define multiple database connections.

Example:

```ts
export default {
  default: "primary",

  connections: {
    primary: {},
    analytics: {}
  }
};
```

Usage:

```ts
Config.get(
  "database.connections.primary"
);
```

This is useful for larger applications.

---

# Cache Configuration

Cache behavior is typically configured through:

```text
config/cache.ts
```

Example:

```ts
export default {
  driver: Env.get(
    "CACHE_DRIVER"
  )
};
```

Possible drivers:

```text
memory

redis

file
```

Different environments can use different cache implementations.

---

# Redis Configuration

Redis settings are usually isolated within their own configuration file.

Example:

```text
config/redis.ts
```

```ts
export default {
  host: Env.get(
    "REDIS_HOST"
  ),

  port: Env.get(
    "REDIS_PORT"
  )
};
```

This configuration may be shared by:

- Cache
- Queues
- Rate limiting
- Realtime systems

---

# Storage Configuration

Storage systems often support multiple disks.

Example:

```text
config/storage.ts
```

```ts
export default {
  default: "local",

  disks: {
    local: {},
    s3: {}
  }
};
```

Applications can switch storage providers without changing business logic.

---

# WebSocket Configuration

# Configuration Loading

During startup, Bejibun loads configuration files.

Lifecycle:

```text
Application Start
         │
         ▼
Load Environment
         │
         ▼
Load Config Files
         │
         ▼
Register Services
         │
         ▼
Application Ready
```

Configuration is available before services are initialized.

---

# Configuration and Service Providers

Service Providers frequently consume configuration.

Example:

```ts
container.singleton(
  Storage,
  () => {
    return new Storage(
      Config.get(
        "storage.default"
      )
    );
  }
);
```

Providers use configuration to construct framework services.

---

# Configuration and Dependency Injection

Configuration services can be injected into classes.

Example:

```ts
export class StorageService {
  constructor(
    protected config:
      Config
  ) {}
}
```

This allows classes to remain configurable without hardcoded values.

---

# Environment-Specific Configuration

One application can behave differently across environments.

Development:

```env
APP_ENV=development
CACHE_DRIVER=memory
```

Production:

```env
APP_ENV=production
CACHE_DRIVER=redis
```

The application code remains identical.

Only configuration changes.


```ts
export default {
  provider: "stripe"

```

Usage:

```ts
Config.get(
```

---

# Configuration Validation

Missing configuration values can cause failures.

Example:

```env
DB_HOST=
```

A configuration validation system can verify required values during startup.

Benefits:

```text
Early Error Detection

Safer Deployments

Predictable Behavior
```

Failing fast is often preferable to runtime failures.

---

# Configuration Caching

Large applications may cache configuration for improved startup performance.

Conceptually:

```text
Read Configuration
         │
         ▼
Generate Cache
         │
         ▼
Load Cached Configuration
```

Benefits:

```text
Faster Startup

Reduced File Reads

Improved Performance
```

# Best Practices

## Use Configuration Files

Good:

```ts
Config.get(
  "database.host"
);
```

Avoid:

```ts
const host =
  "localhost";
```

---

## Keep Secrets in Environment Variables

Good:

```env
DB_PASSWORD=secret
```

Avoid:

```ts
password: "secret"
```

inside configuration files.

---

## Group Related Settings

Good:

```text
database.ts
cache.ts
storage.ts
```

Avoid large configuration files containing unrelated settings.

---

## Use Meaningful Keys

Good:

```ts
storage.default
```

Avoid:

```ts
s
```

Clear naming improves maintainability.

---

## Centralize Configuration

Application settings should have a single source of truth.

---

# Common Mistakes

### Hardcoding Values

Avoid:

```ts
const port = 5432;
```

Use configuration instead.

---

### Accessing Environment Variables Everywhere

Avoid:

```ts
Env.get(...)
```

throughout application code.

Prefer:

```ts
Config.get(...)
```

This improves abstraction.

---

### Storing Secrets in Git

Never commit production credentials.

---

### Mixing Concerns

Keep configuration files focused on configuration only.

Avoid placing business logic inside configuration files.
