# Redis

Modern applications require more than a database.

They often need:

- Fast caching
- Background jobs
- Session storage
- Rate limiting
- Distributed locks
- Real-time communication

While databases excel at permanent storage, they are not always the best choice for high-speed operations.

Redis provides an in-memory data store designed for extremely fast data access.

---

# What Is Redis?

Redis is an in-memory key-value data store.

Unlike traditional databases:

```text
Database
     │
     ▼
Disk Storage
```

Redis primarily operates in memory:

```text
Redis
   │
   ▼
Memory
```

This allows data to be read and written extremely quickly.

---

# Why Redis Exists

Without Redis:

```text
Request
   │
   ▼
Database
   │
   ▼
Response
```

Repeated thousands of times.

Problems:

```text
High Database Load

Slower Responses

Increased Resource Usage
```

Redis helps offload many of these operations.

---

# Redis Philosophy

Not all data needs permanent storage.

Examples:

```text
Cache

Sessions

Temporary Tokens

Job Queues
```

These types of data benefit from fast in-memory access.

Redis specializes in handling them efficiently.

---

# Redis Lifecycle

Typical Redis workflow:

```text
Application
      │
      ▼
Redis
      │
      ▼
Return Data
```

Because Redis stores data in memory, operations are often completed in milliseconds.

---

# Redis As A Cache

One of Redis's most common uses is caching.

Flow:

```text
Request
   │
   ▼
Redis Cache
   │
   ├── Hit
   │      │
   │      ▼
   │   Return Data
   │
   └── Miss
          │
          ▼
       Database
          │
          ▼
       Cache Result
```

This significantly reduces database traffic.

---

# Redis As A Key-Value Store

Redis stores data using keys and values.

Example:

```text
user:1:name
     │
     ▼
John Doe
```

The key identifies the value.

---

# Key Structure

Good Redis keys are descriptive.

Examples:

```text
user:1

user:1:profile

settings:app

dashboard:stats
```

Clear key naming improves maintainability.

---

# String Values

The simplest Redis data type.

Example:

```text
Key:
site:name

Value:
Bejibun
```

Useful for configuration and small values.

---

# Numeric Values

Redis can efficiently store counters.

Examples:

```text
Page Views

Downloads

API Usage
```

Redis excels at high-frequency counting operations.

---

# Expiring Data

Redis supports automatic expiration.

Example:

```text
Store Value
      │
      ▼
Expire After 1 Hour
```

After expiration:

```text
Value Removed Automatically
```

This is ideal for temporary data.

---

# Time To Live (TTL)

TTL determines how long data remains available.

Examples:

```text
60 Seconds

5 Minutes

1 Hour

24 Hours
```

TTL is heavily used for caching.

Lists store ordered collections.

Examples:

---


# Best Practices

## Use Descriptive Keys

Organize keys consistently.

---

## Set Expiration When Appropriate

Temporary data should expire automatically.

---

## Avoid Storing Large Objects

Large values consume memory quickly.

---

## Monitor Memory Usage

Redis performance depends heavily on available memory.

---

## Use Redis For Fast Access

Avoid treating Redis as a primary database.

# Common Mistakes

### Using Redis For Permanent Storage

Redis is primarily designed for speed.

---

### Forgetting TTLs

Temporary data may accumulate unnecessarily.

---

### Poor Key Naming

Unstructured keys become difficult to manage.

---

### Storing Excessive Data

Memory is finite.

---

### Ignoring Monitoring

Production Redis instances should be monitored continuously.
