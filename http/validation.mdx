# Validation

Validation is the process of verifying that incoming data meets the requirements of your application before it is processed.

Every application receives input from external sources:

- HTTP requests
- Forms
- APIs
- Query parameters
- File uploads
- WebSocket messages

Because this data originates outside your application, it should never be trusted without validation.

Validation helps ensure:

- Data integrity
- Application stability
- Security
- Predictable behavior
- Better user experience

---

# Why Validation Matters

Consider a user registration endpoint.

Expected request:

```json
{
  "name": "John Doe",
  "email": "john@example.com",
  "password": "secret123"
}
```

Without validation:

```json
{
  "name": "",
  "email": "not-email",
  "password": "1"
}
```

could enter the system.

This may lead to:

```text
Corrupted Data

Application Errors

Security Risks

Unexpected Behavior
```

Validation prevents invalid data from progressing further into the application.

---

# Validation Lifecycle

Validation occurs before business logic executes.

```text
Request
   │
   ▼
Validation
   │
   ▼
Valid Data
   │
   ▼
Controller
   │
   ▼
Service
```

If validation fails:

```text
Request
   │
   ▼
Validation
   │
   ▼
Validation Error
   │
   ▼
422 Response
```

This keeps invalid data out of the application.

---

# Validation Philosophy

Validation should be:

```text
Declarative

Predictable

Reusable

Readable

Centralized
```

Rather than manually checking values throughout controllers, validation rules should be defined in dedicated validators.

---

# Manual Validation

Validation can be performed manually.

Example:

```ts
if (!email) {
  throw new Error(
    "Email is required"
  );
}
```

However, manual validation quickly becomes repetitive.

Example:

```ts
if (!name) {}

if (!email) {}

if (!password) {}
```

This approach does not scale well.

---

# Validator Classes

Bejibun encourages defining validation rules in dedicated validator classes.

Example:

```ts
export default class CreateUserValidator {
  //
}
```

Benefits:

```text
Reusability

Consistency

Separation of Concerns
```

Validators become a single source of truth for input requirements.

---

# Request Validation

Controllers typically validate requests before executing business logic.

Example:

```ts
const data =
  await request.validate(
    CreateUserValidator
  );
```

Flow:

```text
Request
   │
   ▼
Validator
   │
   ▼
Validated Data
   │
   ▼
Controller
```

The controller receives trusted data.

# Validation Rules

Rules describe what data is allowed.

Examples:

```text
Required

String

Number

Boolean

Email

URL

Date
```

Rules define the expected shape of incoming data.

---

# Required Fields

A required rule ensures a field exists.

Example:

```json
{
  "email": ""
}
```

Validation:

```text
email is required
```

Result:

```http
422 Unprocessable Entity
```

Required validation is one of the most commonly used rules.

# Validation Errors

When validation fails, the framework generates a validation exception.

Example:

```text
Validation Failed
        │
        ▼
Validation Exception
        │
        ▼
422 Response
```

The controller action does not execute.

---

# Validation Responses

Example response:

```json
{
  "errors": {
    "email": [
      "The email field is required."
    ]
  }
}
```

Clients can use this information to display validation feedback.

---

# Multiple Validation Errors

Validation may return multiple errors simultaneously.

Example:

```json
{
  "errors": {
    "name": [
      "Required"
    ],
    "email": [
      "Invalid email"
    ]
  }
}
```

Providing all errors at once improves user experience.

# Validation and Controllers

Controllers should validate before performing work.

Good:

```ts
const data =
  await request.validate(
    CreateUserValidator
  );
```

Avoid:

```ts
await service.create(
  request.body()
);
```

Unvalidated data should not reach business logic.

---

# Best Practices

## Validate All External Input

Never trust incoming data.

---

## Use Dedicated Validators

Keep validation rules separate from controllers.

---

## Return Consistent Errors

Validation failures should follow a predictable format.

---

## Validate Files

Always validate uploaded files.

---

## Keep Rules Declarative

Validation rules should describe requirements clearly.

---

# Common Mistakes

### Skipping Validation

Every external input should be validated.

---

### Duplicating Rules

Keep rules centralized.

---

### Mixing Validation and Business Logic

Validation should verify data.

Business logic should process data.

---

### Returning Vague Errors

Provide meaningful feedback whenever possible.

---

### Trusting Client-Side Validation

Client-side validation improves UX but is never a substitute for server-side validation.

✔ All external data should be considered untrusted

✔ Validators centralize validation rules

✔ Validation occurs before business logic executes
