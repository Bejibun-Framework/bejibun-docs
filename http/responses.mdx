# Responses

A response is the data sent back to the client after a request has been processed.

Every request handled by a Bejibun application eventually produces a response.

The Response system is responsible for:

- Returning data
- Sending JSON
- Managing status codes
- Setting headers
- Sending cookies
- Delivering files
- Redirecting clients
- Standardizing API output

Responses are the final stage of the request lifecycle.

---

# What Is a Response?

An HTTP response is the server's answer to a request.

Example request:

```http
GET /users/1
```

Response:

```json
{
  "id": 1,
  "name": "John"
}
```

The client receives this data and acts accordingly.

---

# Response Lifecycle

Every request eventually becomes a response.

```text
Request
   │
   ▼
Router
   │
   ▼
Controller
   │
   ▼
Response
   │
   ▼
Client
```

The Response object represents the final output of the application.

# Returning Simple Data

Controllers can return simple values.

Example:

```ts
async index() {
  return "Hello World";
}
```

Response:

```text
Hello World
```

The framework converts the return value into a valid HTTP response.

# Returning Objects

Objects are commonly returned from API endpoints.

Example:

```ts

  return {
    id: 1,
    name: "John"
  };
}
```

Response:

```json
{
  "id": 1,
  "name": "John"
}
```

Objects are automatically serialized.

# JSON Responses

JSON is the most common response type in modern APIs.

Example:

```ts
return Response.json({
  success: true
});
```

Response:

```json
{
  "success": true
}
```

JSON responses provide a consistent format for API consumers.

# HTTP Status Codes

Every response contains a status code.

Example:

```http
200 OK
```

Status codes communicate the outcome of an operation.

---

# Common Success Codes

| Status | Meaning |
|----------|----------|
| 200 | OK |
| 201 | Created |
| 202 | Accepted |
| 204 | No Content |

Example:

```ts
return Response.json(
  data,
  201
);
```

Response:

```http
201 Created
```

---

# Common Error Codes

| Status | Meaning |
|----------|----------|
| 400 | Bad Request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Not Found |
| 409 | Conflict |
| 422 | Validation Error |
| 429 | Too Many Requests |
| 500 | Internal Server Error |

These codes help clients understand failures.

---

# Custom Status Codes

Example:

```ts
return Response.json(
  {
    message: "Created"
  },
  201
);
```

Result:

```http
201 Created
```

Controllers should return meaningful status codes whenever possible.

# Response Headers

Headers provide metadata about a response.

Example:

```ts
Response.header(
  "X-Version",
  "1.0"
);
```

Result:

```http
X-Version: 1.0
```

Headers are commonly used for:

- Authentication
- Caching
- Content negotiation
- Security

---

# Multiple Headers

Example:

```ts
Response.headers({
  "X-Version": "1.0",
  "X-API": "Bejibun"
});
```

Response headers become part of the outgoing response.

# Best Practices

## Use Appropriate Status Codes

Good:

```http
201 Created
```

for resource creation.

Avoid:

```http
200 OK
```

for every situation.

---

## Keep Responses Consistent

Clients should always know what to expect.

---

## Return Structured Errors

Error responses should follow a predictable format.

---

## Use JSON For APIs

JSON remains the standard for API communication.

---

## Protect Sensitive Data

Never expose:

```text
Passwords

Tokens

Secrets

Internal Metadata
```

---

# Common Mistakes

### Inconsistent Response Formats

Avoid returning completely different structures across endpoints.

---

### Incorrect Status Codes

Status codes should reflect the actual result.

---

### Exposing Internal Errors

Production responses should not expose stack traces.

---

### Returning Excessive Data

Only return data clients actually need.

---

### Ignoring Caching

Many responses can benefit from cache headers.
