# Requests

Every interaction with a Bejibun application begins with an HTTP request.

When a client sends a request, Bejibun converts it into a Request object that provides a clean and consistent API for accessing incoming data.

The Request system is responsible for:

- Reading request data
- Accessing query parameters
- Reading route parameters
- Accessing headers
- Managing cookies
- Handling file uploads
- Retrieving client information
- Validating input

Understanding requests is fundamental because every controller action receives data through the Request object.

---

# What Is a Request?

An HTTP request represents a client's attempt to interact with your application.

Example:

```http
POST /users

Content-Type: application/json
```

Body:

```json
{
  "name": "John",
  "email": "john@example.com"
}
```

The Request object provides access to this information in a structured way.

---

# Request Lifecycle

Every request follows a predictable path.

```text
Client
   │
   ▼
HTTP Request
   │
   ▼
Router
   │
   ▼
Middleware
   │
   ▼
Controller
   │
   ▼
Response
```

The Request object exists throughout this lifecycle.

---

# Accessing The Request

Controllers typically receive access to the current request.

Example:

```ts
export default class UserController {
  async store(request: Request) {
    //
  }
}
```

The framework automatically provides the current Request instance.

---

# Request Structure

A request contains multiple types of data.

```text
Request
   │
   ├── URL
   ├── Method
   ├── Headers
   ├── Query Parameters
   ├── Route Parameters
   ├── Cookies
   ├── Files
   └── Body
```

All of this information is accessible through the Request API.

---

# Request Method

The request method indicates the type of operation being performed.

Example:

```http
GET
POST
PUT
PATCH
DELETE
```

Accessing the method:

```ts
request.method();
```

Possible result:

```text
POST
```

---

# Request URL

The full request URL is available.

Example:

```ts
request.url();
```

Result:

```text
https://example.com/users
```

This can be useful for logging and debugging.

---

# Request Path

Sometimes only the path is needed.

Example:

```ts
request.path();
```

Result:

```text
/users
```

The path excludes the domain and query string.

---

# Request Headers

Headers provide metadata about the request.

Example:

```http
Authorization: Bearer token

Content-Type: application/json
```

Accessing a header:

```ts
request.header(
  "authorization"
);
```

Result:

```text
Bearer token
```

Headers are commonly used for authentication and content negotiation.

---

# Checking Headers

Example:

```ts
request.hasHeader(
  "authorization"
);
```

Result:

```ts
true
```

This allows conditional behavior based on header presence.

---

# Request Body

The request body contains submitted data.

Example request:

```json
{
  "name": "John",
  "email": "john@example.com"
}
```

Accessing data:

```ts
request.body();
```

Result:

```json
{
  "name": "John",
  "email": "john@example.com"
}
```

---

# Retrieving Input

Specific fields can be accessed directly.

Example:

```ts
request.input("name");
```

Result:

```text
John
```

This is often more convenient than reading the entire body.

---

# Default Values

Missing fields can return fallback values.

Example:

```ts
request.input(
  "role",
  "user"
);
```

Result:

```text
user
```

This helps avoid unnecessary conditional checks.

---

# Query Parameters

Query parameters appear in the URL.

Request:

```http
GET /users?page=2
```

Accessing:

```ts
request.query("page");
```

Result:

```text
2
```

Query parameters are commonly used for:

- Pagination
- Filtering
- Searching
- Sorting

---

# Retrieving All Query Parameters

Example:

```ts
request.query();
```

Result:

```json
{
  "page": "2",
  "limit": "20"
}
```

This returns the complete query object.

---

# Route Parameters

Route parameters originate from route definitions.

Route:

```ts
Router.get(
  "/users/:id"
);
```

Request:

```http
GET /users/10
```

Accessing:

```ts
request.param("id");
```

Result:

```text
10
```

These values are extracted by the router.

---

# Retrieving All Parameters

Example:

```ts
request.params();
```

Result:

```json
{
  "id": "10"
}
```

Useful when multiple parameters exist.

---

Example:

Accessing:

# Authentication Data

Authenticated requests often include user information.

Example:

```ts

  id: 1,
```

# Request Validation

Validation is often performed directly from the request.

Example:

```ts
await request.validate(
  CreateUserValidator
);
```

Flow:

```text
Request
    │
    ▼
Validation
    │
    ▼
Controller Logic
```

Validation should occur before processing business logic.

# Request and Controllers

Controllers use the Request object to access client data.

Example:

```ts
async store(
  request: Request
) {
  const email =
    request.input(
      "email"
    );
}
```

The Request object acts as the controller's primary source of input.

---

# Request and Validation

Validation typically occurs before business logic.

Example:

```ts
const data =
  await request.validate(
    CreateUserValidator
  );
```

Benefits:

```text
Clean Controllers

Consistent Validation

Reliable Data
```

# Best Practices

## Validate Early

Validate data before processing it.

---

## Use Typed Validation

Prefer validated data over raw input.

---

## Avoid Directly Trusting Input

All user input should be considered untrusted.

---

## Keep Controllers Clean

Retrieve data from the request and delegate logic to services.

---

## Use Request Helpers

Prefer:

```ts
request.input(...)
```

over manual parsing.

---

# Common Mistakes

### Skipping Validation

Never assume incoming data is valid.

---

### Trusting Client Data

Always validate and sanitize user input.

---

### Reading Raw Data Everywhere

Use the Request API instead of manually parsing requests.

---

### Mixing Business Logic

The Request object should provide data, not contain business rules.

---

### Ignoring Uploaded File Validation

Always validate uploaded files before storing them.

# Key Takeaways

✔ Every incoming request becomes a Request object

✔ Requests provide access to headers, body data, parameters, and files

✔ Query and route parameters are handled separately

✔ Uploaded files are managed through the Request API

✔ Validation should occur before business logic

✔ Middleware and controllers share the same request instance

✔ Request objects provide a consistent developer experience

Understanding Requests is essential because every controller action depends on data arriving through the Request system.
