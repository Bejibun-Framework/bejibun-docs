# Routing

Routing is the mechanism that connects incoming requests to your application logic.

Every HTTP request entering a Bejibun application passes through the router before reaching controllers, middleware, or services.

The routing system is responsible for:

- Matching incoming URLs
- Resolving controllers
- Executing middleware
- Handling route parameters
- Grouping routes
- Applying prefixes
- Managing route names
- Generating URLs

Without routing, the framework would have no way to determine which code should handle a request.

---

# What Is a Route?

A route defines how an application responds to a specific HTTP request.

Example:

```ts
Router.get("/", () => {
  return {
    message: "Hello World"
  };
});
```

Request:

```http
GET /
```

Response:

```json
{
  "message": "Hello World"
}
```

The route tells the framework what to execute when a request matches.

---

# Routing Lifecycle

Every request follows the routing pipeline.

```text
Incoming Request
        │
        ▼
Router
        │
        ▼
Route Match
        │
        ▼
Middleware
        │
        ▼
Controller Action
        │
        ▼
Response
```

This lifecycle powers every endpoint in the application.

# Basic Routes

The simplest route responds to a request.

Example:

```ts
Router.get("/", () => {
  return "Welcome";
});
```

When a request reaches:

```http
GET /
```

the callback is executed.

# Supported HTTP Methods

Bejibun supports all standard HTTP methods.

Example:

```ts
Router.get(...)

Router.post(...)

Router.put(...)

Router.patch(...)

Router.delete(...)
```

Each method serves a different purpose.

| Method | Purpose |
|----------|----------|
| GET | Retrieve data |
| POST | Create resources |
| PUT | Replace resources |
| PATCH | Update resources |
| DELETE | Remove resources |

---

# Route Callbacks

Routes may execute inline callbacks.

Example:

```ts
Router.get("/health", () => {
  return {
    status: "ok"
  };
});
```

This is useful for:

- Health checks
- Prototypes
- Simple endpoints

For larger applications, controllers are preferred.

---

# Controller Routes

Most applications delegate requests to controllers.

Example:

```ts
Router.get(
  "/users",
  [UserController, "index"]
);
```

Controller:

```ts
export default class UserController {
  async index() {
    //
  }
}
```

Benefits:

```text
Separation of Concerns

Reusability

Cleaner Route Files
```

---

# Route Parameters

Routes can capture dynamic values.

Example:

```ts
Router.get(
  "/users/:id",
  [UserController, "show"]
);
```

Request:

```http
GET /users/10
```

Captured value:

```text
id = 10
```

This allows routes to work with dynamic resources.

---

# Multiple Parameters

Routes may capture multiple values.

Example:

```ts
Router.get(
  "/users/:userId/posts/:postId"
);
```

Request:

```http
GET /users/1/posts/20
```

Result:

```text
userId = 1

postId = 20
```

---

# Optional Parameters

Some parameters may be optional.

Example:

```ts
Router.get(
  "/posts/:slug?"
);
```

Valid URLs:

```http
/posts

/posts/my-post
```

Optional parameters increase route flexibility.

---

# Route Constraints

Parameters may be restricted to specific formats.

Example:

```ts
Router.get(
  "/users/:id"
);
```

Constraint:

```text
id must be numeric
```

Benefits:

```text
Improved Validation

Cleaner URLs

Predictable Routing
```

---

# Route Names

Routes can be assigned unique names.

Example:

```ts
Router.get(
  "/users",
  [UserController, "index"]
).name("users.index");
```

Benefits:

```text
URL Generation

Maintainability

Decoupling
```

Named routes allow URLs to change without affecting application code.

---

# URL Generation

Named routes can generate URLs.

Example:

```ts
route("users.index");
```

Result:

```text
/users
```

This avoids hardcoding URLs.

---

# Route Groups

Large applications often organize routes into groups.

Example:

```ts
Router.group(() => {
  //
});
```

Benefits:

```text
Shared Middleware

Shared Prefixes

Cleaner Organization
```

Groups help manage large route collections.

---

# Route Prefixes

A group can share a common URI prefix.

Example:

```ts
Router.group(() => {
  Router.get("/users");
})
.prefix("/api");
```

Generated routes:

```http
/api/users
```

This eliminates repetitive prefixes.

---

# Nested Prefixes

Prefixes can be combined.

Example:

```text
/api
    │
    ▼
/v1
```

Result:

```http
/api/v1/users
```

Useful for API versioning.

---

# Middleware

Routes can apply middleware.

Example:

```ts
Router.get(
  "/profile",
  [ProfileController, "show"]
).middleware([
  AuthMiddleware
]);
```

Flow:

```text
Request
    │
    ▼
Middleware
    │
    ▼
Controller
```

Middleware runs before the controller action.

---

# Multiple Middleware

Several middleware can be applied simultaneously.

Example:

```ts
.middleware([
  AuthMiddleware,
  RateLimitMiddleware
]);
```

Execution:

```text
Auth
  │
  ▼
Rate Limit
  │
  ▼
Controller
```

---

# Group Middleware

Middleware can be applied to an entire route group.

Example:

```ts
Router.group(() => {
  //
})
.middleware([
  AuthMiddleware
]);
```

All routes within the group inherit the middleware.

---

# API Versioning

Versioning allows APIs to evolve safely.

Example:

```http
/api/v1/users

/api/v2/users
```

Route structure:

```text
API
 │
 ├── v1
 │
 └── v2
```

Versioning helps maintain backward compatibility.

# Route Resolution

When a request arrives:

```http
GET /users/10
```

The router:

```text
Find Matching Route
         │
         ▼
Extract Parameters
         │
         ▼
Execute Middleware
         │
         ▼
Resolve Controller
         │
         ▼
Return Response
```

This process occurs automatically.

# RESTful Design

Bejibun encourages RESTful route design.

Good:

```http
GET /users

GET /users/10

POST /users
```

Avoid:

```http
GET /getUsers

POST /createUser
```

Resources should be represented through nouns rather than actions.

# Best Practices

## Keep Routes Thin

Good:

```ts
Router.get(
  "/users",
  [UserController, "index"]
);
```

Avoid placing large amounts of logic inside route definitions.

---

## Use Controllers

Controllers improve organization and maintainability.

---

## Group Related Routes

Good:

```text
Authentication

Users

Posts

Payments
```

Keep routes organized by domain.

---

## Use Route Names

Named routes improve maintainability.

---

## Version Public APIs

Versioning prevents breaking existing clients.

---

# Common Mistakes

### Business Logic in Routes

Avoid:

```ts
Router.get("/", () => {
  // hundreds of lines
});
```

Use controllers instead.

---

### Inconsistent Naming

Choose a clear naming convention and follow it consistently.

---

### Deeply Nested URLs

Avoid overly complex route structures.

---

### Missing Middleware

Protect sensitive routes with appropriate middleware.

---

### Hardcoded URLs

Prefer named routes whenever possible.
